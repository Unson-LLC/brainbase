# 佐藤スタイル意思決定プロトコル v0.1（PO/PM向け）

目的：佐藤に聞かなくても 80%以上の決定が自走する状態を作る。価値観・優先順位の原典は `00_common/knowledge/principles.md` を参照すること。

## 基本原則（優先順）
1. スピード > 完璧（80%確信で決め、走りながら修正）
2. 可逆な決定は即決する（後から変えられるなら悩まない）
3. 学習量が最大化される選択を取る
4. インパクト / 実行難易度 / リードタイムの3軸で判断
5. 言語化までが決定（1〜3行で説明できない決定は不合格）

## 標準フロー（7ステップ）
1) 目的を一文で書く
2) 制約条件を書く（期日・工数・技術・ビジネス）
3) 選択肢を2〜3個出す（A:延長線、B:構造変更、C:やらない/後ろ倒し）
4) 各選択肢を3軸スコアリング（Impact / Effort / Time を1〜5でざっくり）
5) 原則に照らして1つに決める（可逆=学び優先、不可逆=慎重）
6) Next 3 Actions に落とす（誰が / いつまでに / 何を x3）
7) 決定ログに残す（テンプレ使用）

### 決定ログテンプレ
```
# Decision Log

- Date: YYYY-MM-DD
- Project: (Aitle / SalesTailor / Zeims / ...)
- Decision Owner:
- Decision Type: (機能仕様 / UX / 技術構成 / ビジネス条件 / オペレーション)

## 1. Goal
- 一文で

## 2. Constraints
- 期日:
- 工数:
- 技術:
- ビジネス:

## 3. Options
- Option A: 内容 / Pros / Cons
- Option B: 内容 / Pros / Cons
- Option C: (あれば)

## 4. Evaluation
- A: Impact ?, Effort ?, Time ?
- B: Impact ?, Effort ?, Time ?
- C: Impact ?, Effort ?, Time ?

## 5. Decision
- 採用: A / B / C
- 理由（1〜3行）

## 6. Next 3 Actions
- [ ] Action1: 誰が / いつまでに / 何を
- [ ] Action2:
- [ ] Action3:

## 7. Checkpoint
- 日付 or イベント（例：βリリース1週間後に再評価）
```

## エスカレーション基準（佐藤に上げるとき）
上げるのは以下のいずれかのみ：
- ビジネス影響大：月次売上に +/-100万円以上、主要顧客の解約・信頼毀損リスク
- 不可逆な技術決定：アーキ大変更、メインDB/インフラ/認証基盤の切替、既存大部分の書き換え前提
- 組織・契約・株主への影響：株主構成、契約形態、レベニューシェア、長期/独占契約

エスカレーション方法：
- 上記テンプレで Decision Log を書き、Slackで「結論＋レビュー依頼」として送る。
- 例:「自分はBを採用します。異論あればフィードバックください。」

## 典型パターン別指針
- 機能追加 vs 既存改善：原則「既存磨き込み」優先（学びが新機能側に明確なら例外）
- ユーザー要望：単発対応は原則やらない。繰り返すパターンは構造的に解決。
- 技術的に綺麗か vs 早く出せるか：可逆なら速度優先。不可逆なら将来制約を佐藤に相談。

## 運用ルール
- PO/PMは週1件以上 Decision Log を残す。
- 週次定例で新規Decisionレビューと1〜2件の再評価を必ず実施。
- ログなしで「なんとなく進む」は禁止。
