# codex ディレクトリの使い方（運用ハブ）

目的：全事業共通の「意思決定・運用ドキュメント」の一元置き場。コードや環境構築手順は各プロジェクト側に置き、ここでは構造・ルール・テンプレートだけを管理する。

## フォルダ一覧と用途
- `00_common/` 共通ルール・テンプレ・KPI定義。最初にここを参照。
- `00_common/decision_protocol.md` 佐藤スタイル意思決定プロトコル（PO/PMが自走するための型）。
- `00_common/knowledge/` 意思決定ナレッジ庫（前提・定石・過去Decisionの再利用パターン）。
- `meta/` 人・組織タグ・顧客/契約・RACIの正本（責任/契約ラインの一意表）。
- `zeims/` Zeims 向けの戦略・オファー・営業/CS手順・KPI定義。
- `aitle/` Aitle 向けの戦略・オファー・営業/CS手順・KPI定義。
- `techknight/` TechKnight（必要時に同型で展開）。
- `ops/` GM/AE/CSの横断プレイブック、会議リズム、ダッシュボード定義。
- `inbox/` 未整理メモの仮置き。週1で各フォルダへ仕分け。

## 運用ルール（更新の仕方）
1. 新しい事業を追加するときは、`codex/` に事業フォルダを作り、この README の「フォルダ一覧」に1行追記する。
2. 共通で使うテンプレート・ルールは必ず `00_common/` へ集約（各リポに分散させない）。
3. フォルダ目的が変わったら、変更と同時にこの README を更新する（単一の最新版を維持）。
4. `inbox/` は週1で空にする（仕分け or 廃棄）。溜めない。

## プロジェクト側との分担
- `codex/`: ルール・手順・KPI・テンプレ（運用/ビジネス文書）。
- 各プロジェクトフォルダ（例: `zeims/`, `salestailor/` 等）: 実装コード、技術設計、環境設定、リポ固有のREADME/AGENTS。

## メンテ頻度の目安
- 週次：`rules.md` と KPI 定義の整合チェック（会議前に更新）。
- 月次：事業別戦略メモ（01_strategy.md）の現状反映。
- 四半期：フォルダ一覧と役割の棚卸し。
