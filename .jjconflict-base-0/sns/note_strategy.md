# note戦略（さとけい｜AIにマネジメントを丸投げする男）

最終更新: 2026-01-04

---

## 1. ミッション

**noteの役割**: Xで「点」として発信した内容を「線」として繋ぎ、深掘りして資産化する

| プラットフォーム | 役割 | 比喩 |
|-----------------|------|------|
| X | 認知・拡散・日常接触 | チラシ・立ち話 |
| note | 信頼構築・AIO・資産化 | 書籍・講演 |

---

## 2. 狙い（Why note?）

### 2.1 AIO（AI検索最適化）

**目的**: ChatGPT/Perplexity/Google AI Overview に参照される

noteが強い理由:
- 見出し構造が標準装備 → AIが解析しやすい
- Googleと資本業務提携（2025年）→ ドメイン評価が高い
- 専門性の高いコンテンツが多い → AIの信頼性判定で有利

**ターゲットクエリ例**:
- 「AIエージェント マネジメント 自動化」
- 「1人経営 複数事業 運営方法」
- 「Claude Code 実装 経験」
- 「事業OS 設計」

### 2.2 検索流入（SEO）

Xはヒットしにくいが、noteはGoogle/Yahoo!で上位表示される

**狙うキーワード**:
| カテゴリ | キーワード例 |
|---------|-------------|
| AI実装 | Claude Code 実装, MCP 活用, AIエージェント 業務自動化 |
| 経営 | 1人経営 複数事業, マネジメント 自動化, 事業OS |
| 実験記録 | AI 業務委譲, 自動化率 |

### 2.3 ブランディング

Xでは伝えきれない「想い」「判断の理由」「過程」を深掘り

**効果**:
- 「この人に任せて大丈夫そう」という信頼構築
- 合わない顧客の事前フィルタ（価値観の明文化）
- ファン層・ロイヤルユーザーの育成

## 3. コンテンツの柱（5テーマ）

軸足（x_account_profile.md）に基づいて設計

### 柱1: マネジメント自動化の実験記録

**概要**: 「AIにマネジメントを丸投げする」実験の週次レポート

**内容例**:
- 今週の自動化率（68%→72%）
- 何を自動化したか（会議録のSlack自動投稿等）
- うまくいかなかったこと
- 次週の計画

**検索キーワード**: AIエージェント マネジメント, 業務自動化 実験

### 柱2: AIエージェント実装の舞台裏

**概要**: Claude Code/MCP/mana等の実装経験を技術的に深掘り

**内容例**:
- mana-weeklyの半自動化設計（なぜ完全自動化しなかったか）
- brainbaseのアーキテクチャ決定過程
- 失敗した実装アプローチ

**検索キーワード**: Claude Code 実装, MCP 活用, AIエージェント 開発

### 柱3: 事業OS設計の判断と理由

**概要**: 複数事業を統一構造で回す仕組みの設計思想

**内容例**:
- RACI運用の導入経緯
- タスク管理を一本化した理由
- 「情報の一本化」の4大原則

**検索キーワード**: 事業OS, 複数事業 運営, RACI 活用

### 柱4: 失敗と学び

**概要**: 赤裸々な失敗談と、そこから得た教訓

**内容例**:
- 完全自動化を目指して失敗した話
- ツール選定で間違えた話
- 採用・組織で失敗した話

**検索キーワード**: スタートアップ 失敗談, 経営 学び

### 柱5: Xで反応の良かった話題の深掘り

**概要**: Xでバズった投稿を、背景・理由・具体例を交えて展開

**プロセス**:
```
X投稿がバズる（1万imp以上）
    ↓
「なぜこれがウケたか」分析
    ↓
背景・判断理由・具体例を追加してnote化
    ↓
noteをXでシェア
```

## 4. 記事テンプレート

### 4.1 基本構成

```markdown
# タイトル（検索キーワードを含む、30文字以内）

## 導入（フック）
- 結論を最初に提示
- 読者の「自分ごと」化
- 200文字以内

## 本文
### 見出し1（悩み・課題）
- 何に悩んだか
- なぜそれが問題だったか

### 見出し2（判断・選択）
- どういう選択肢があったか
- なぜその判断をしたか

### 見出し3（結果・学び）
- どうなったか（数字があれば数字で）
- 何を学んだか

## まとめ
- 3行で要約
- 問いかけで締める（「みんなはどうしてる？」等）

## CTA
- Xフォロー誘導
- 関連記事リンク
```

### 4.2 タイトルの型

| 型 | 例 |
|---|---|
| 数字＋結果 | 「自動化率を68%→72%にした3つの施策」 |
| 失敗→学び | 「完全自動化を目指して失敗した話」 |
| 常識破壊 | 「マネジメントを人間がやる時代、終わらせたい」 |
| How to | 「1人で7事業を回すための事業OS設計」 |
| Before/After | 「タスク管理を一本化したら何が変わったか」 |

### 4.3 見出しの型

AIO対策として、以下の形式を推奨:

- 「〜とは？」（定義）
- 「〜の理由」（解説）
- 「〜の方法」（手順）
- 「〜のメリット/デメリット」（比較）
- 「〜の結果」（実績）

## 5. スタイルガイド（X との違い）

### 5.1 許可される表現（noteのみ）

| 項目 | X | note |
|------|---|------|
| 末尾の句点 | ❌ 禁止 | ✅ OK（読みやすさ重視） |
| 箇条書き | ❌ 禁止 | ✅ OK（整理しやすさ重視） |
| 見出し | 不要 | ✅ 必須（AIO対策） |
| 文字数 | 280文字 | 1,500〜3,000文字 |
| Markdown | 最小限 | ✅ 活用OK |

### 5.2 共通ルール（X・note共通）

| 項目 | ルール |
|------|--------|
| 一人称 | 俺（カジュアル）、自分（客観的）、私（フォーマル） |
| 口調 | 〜だよね、〜なんよな、〜っすね |
| 感情表現 | マジで、やばい、うぉー！ |
| 禁止 | 硬い言葉遣い、優等生発信、一般論だけ |

### 5.3 軸足チェック（必須）

エッジの効いた発言をする前に確認:

- [ ] このトピックは専門領域か？（x_account_profile.md参照）
- [ ] 経験・実績の裏付けがあるか？
- [ ] 反論された場合、根拠を示せるか？

**専門領域**:
- ✅ 事業OS設計、自動化設計、AIツール実装
- ✅ PM業務自動化、タスク管理・RACI運用
- ✅ 意思決定プロセス、マネジメント自動化

**軸足の外**:
- ❌ AI研究・最先端技術、特定業界の専門知識
- ❌ 大企業組織論、学術的な理論

## 6. AIO対策（AI検索最適化）

### 6.1 記事構造

- [ ] タイトルに検索キーワードを含む
- [ ] 導入200文字以内で結論を提示
- [ ] 見出し（h2, h3）を適切に使用
- [ ] 見出しに検索キーワードを含む
- [ ] 箇条書きで情報を整理
- [ ] 具体的な数字・実績を含む

### 6.2 コンテンツ品質

- [ ] 専門性が高い内容（軸足の範囲内）
- [ ] オリジナルの経験・データを含む
- [ ] 悩み→判断→結果の過程を明示
- [ ] 読者の疑問に先回りして回答

### 6.3 更新頻度

- 週1-2本の新規記事
- 過去記事の定期更新（3ヶ月に1回見直し）

## 7. 運用ルール

### 7.1 投稿頻度

| 頻度 | 内容 |
|------|------|
| 週1 | 必須: マネジメント自動化の週次レポート |
| 週0-1 | 任意: 技術深掘り or 失敗談 or Xバズ深掘り |

### 7.2 投稿フロー

```
1. Xで反応を観察（バズった話題をストック）
    ↓
2. 週末に記事テーマを決定
    ↓
3. テンプレートに沿って執筆（1,500〜3,000文字）
    ↓
4. AIOチェックリスト確認
    ↓
5. 投稿
    ↓
6. Xでシェア
```

### 7.3 Xとの連携

| タイミング | アクション |
|-----------|-----------|
| note投稿時 | Xでシェア（要約＋リンク） |
| Xバズ時 | note深掘り記事のネタとしてストック |
| 週次 | note投稿をX固定ポストに追加検討 |

## 9. 禁止事項

### 9.1 コンテンツ

- ❌ 軸足から外れた断定・批判
- ❌ 一般論だけの記事（必ず自分の経験を含める）
- ❌ 結果だけの報告（悩み→判断→結果の過程を含める）
- ❌ 優等生発信（無難な内容）

### 9.2 運用

- ❌ 週1回未満の投稿（継続性が重要）
- ❌ Xとの連携なし（相乗効果を逃す）
- ❌ 見出しなしの記事（AIO対策必須）
