## ユーザーストーリー

**誰が**: 配下の AI 作業者群を監督し、事業責任者への上申責任も持つ現場責任者  
**何を**: `停滞中の施策` を起点に、短い壁打ち、再割り振り、証跡確認、事業責任者向け上申準備までを 1 本の **標準運用導線** として回したい
**なぜ**: `即応パネル` や `論点整理室` を、名前だけの UI ではなく、誰が使っても同じ判断品質に寄せられる v0 の運用型へ落としたいから

## 背景

- 既存 OSS にも board、live feed、approval dashboard、assistant drawer はある
- したがって UnsonOS の差別化は、`現場責任者の判断面` が本当に作業者監督や証跡主体の上申へつながるかで決まる
- `即応壁打ち = 即応パネル`
- `深掘り壁打ち = 論点整理室`
- `生セッション / 生チャット = 詳細 / Brainbase`
  という整理は固まってきたが、まだ v0 の標準手順、チェックリスト、完了条件としては固定されていない
- 会話記録は保持するが、主成果物にしてはいけない
- `VibePro` 的に必要なのは、概念だけでなく **誰がやっても同じように回る最小の運用型** である

## v0 で固定する導線

### 対象フロー

`停滞中の施策 -> 状況要約 -> 証跡確認 -> 成果物要求 or 再割り振り -> 上申資料 -> 事業責任者の判断待ち一覧`

### 期待する現場責任者の動き

1. 現場責任者が `判断待ち一覧` または `進行レーン` で停滞中の施策を見つける
2. `即応パネル` で即応壁打ちを行い、現状要約とボトルネック把握を得る
3. `証跡確認` で不足証跡を確認する
4. `割り振り支援` で経路変更候補や配分変更案を見る
5. その場で `成果物要求` または `再割り振り` を行う
6. 必要時のみ `論点整理室` を開き、事業責任者へ上げる論点と反対意見を整理する
7. 最後に `上申資料` か `判断メモ` を作成して、事業責任者の `判断待ち一覧` に送る

## v0 の標準手順

### 1. 朝の確認

- `区画ヘッダー` で `停滞中 / 承認待ち / 成果物不足 / 昨日からの変化` を確認する
- `判断待ち一覧` で本日先に裁く施策を 3 件以内に絞る
- この時点では `詳細作業面` に降りない

### 2. 介入

- 停滞中の施策を 1 件選ぶ
- `即応パネル` で `状況要約 -> 証跡確認 -> 割り振り支援` の順で見る
- `成果物要求` か `再割り振り` のどちらか 1 つを必ず確定する

### 3. 上申判定

- 次のどれかに当てはまる場合だけ `論点整理室` に進む
  - 選択肢が 2 つ以上競合している
  - 他施策への影響がある
  - 事業責任者の承認が必要
  - 停止 / 縮小 / 継続を比べる必要がある

### 4. 上申

- `論点整理室` では `進める案 / 縮小案 / 停止案` を整理する
- 出口は `上申資料` または `判断メモ` のどちらかに固定する
- `上申資料` は事業責任者の `判断待ち一覧` に載る形で閉じる

### 5. 夕方の締め

- `今日裁いた判断`
- `今日出した成果物要求`
- `今日変えた割り振り`
- `上申中`
- `明日に持ち越す停滞`

を同じ区画画面で確認する

## v0 の最小チェックリスト

### 朝の確認チェック

- 今日先に裁く施策が 3 件以内に絞られている
- `停滞中 / 承認待ち / 成果物不足` のどれが主因か見えている
- 生ログを読まなくても介入候補が決まっている

### 介入前チェック

- 主因が 1 つ以上特定されている
- 足りない証跡が列挙されている
- `成果物要求` と `再割り振り` のどちらを先に打つか決まっている

### 上申前チェック

- `進める案 / 縮小案 / 停止案` が並んでいる
- 推奨案が 1 つに絞られている
- 足りない証跡と影響範囲が書かれている
- 事業責任者が 5 分で裁ける粒度になっている

### 夕方の締めチェック

- 今日の判断が `判断メモ` または `上申資料` に変換されている
- 未解決の停滞が明日の介入候補として残っている
- `詳細作業面` に落ちた案件が何件あったか把握できる

## v0 で使う最小成果物

- `証跡確認票`
  - 不足証跡
  - 要求先
  - 期限
  - 関連施策
- `上申資料`
  - 現在状態
  - 進める案 / 縮小案 / 停止案
  - 推奨案
  - 影響範囲
  - 足りない証跡
  - 現場責任者コメント
- `判断メモ`
  - 裁いた内容
  - 判断理由
  - 参照証跡
  - 次回見直し条件

## 目指す状態

- 現場責任者が作業者の生セッションを毎回見なくても、管理者として判断に必要な情報へ到達できる
- 即応壁打ちだけで済むケースと、深掘り壁打ちが必要なケースが分離される
- 再割り振り、成果物要求、事業責任者への上申が 1 本の導線に収まる
- 会話記録は下層保持されるが、上位の主成果物は `判断メモ / 上申資料 / 判断理由` になる
- 現場責任者が作業者監督者ではなく、管理者 / 統制役のままでいられる
- 同じ導線を別の現場責任者が踏んでも、チェックリストと成果物の型が揃う

## v0 でやらないこと

- 経営者まで含む全役割の end-to-end 運用
- 裁量昇格候補の本格運用
- 全進行段階の DoD 定義
- runtime 差異を吸収した完全な実装仕様
- Slack や Gmail を置き換えること

## 受け入れ条件

| AC | 内容 | 検証方法 |
|---|---|---|
| AC1 | v0 対象フローが `停滞中の施策 -> 状況要約 -> 証跡確認 -> 成果物要求 or 再割り振り -> 上申資料 -> 事業責任者の判断待ち一覧` の順で明文化される | ストーリーを読んで入口と出口が一意に辿れることを確認 |
| AC2 | 即応壁打ちが `即応パネル`、深掘り壁打ちが `論点整理室` として分離される | それぞれの役割が別の面として書かれていることを確認 |
| AC3 | `割り振り支援` と `証跡確認` が、会話ではなく操作と判断オブジェクトに接続される | 再割り振り / 成果物要求 / 上申資料への出口があることを確認 |
| AC4 | 現場責任者が生セッション / 生チャットを常時見なくても裁ける前提になっている | `詳細 / Brainbase` が退避面に下げられていることを確認 |
| AC5 | 会話記録が保持される前提は残しつつ、主成果物は `判断メモ / 上申資料 / 判断理由` になる | 会話記録を捨てず、ただし上位成果物にしないことが明記されることを確認 |
| AC6 | 朝の確認、介入前、上申前、夕方の締めの 4 つのチェックリストがある | ストーリーを読んで運用の型が一意に辿れることを確認 |
| AC7 | `証跡確認票 / 上申資料 / 判断メモ` の最小成果物が定義されている | ストーリーを読んで運用成果物が決まっていることを確認 |
| AC8 | この導線が失敗する条件として、現場責任者の作業者化、即応 / 深掘りの使い分け崩壊、操作不在が挙げられている | リスク節に失敗条件が書かれていることを確認 |

## 主な失敗条件

- `即応パネル` がアシスタント引き出しに逆流し、長文チャット置き場になる
- `論点整理室` が空のチャットから始まり、毎回文脈説明が必要になる
- `割り振り支援` が提案だけ返して操作プレビューを持たない
- `証跡確認` が会話記録読み込みの代替にならず、結局生セッションへ戻る
- `事業責任者への上申` が資料ではなく会話ログ転送に終わる
- 現場責任者が作業者一人ずつへ個別指示を出し始める
- 同じ介入を別の現場責任者が再現できず、毎回やり方がぶれる

## 優先度

**P0** - 現場責任者コンソールが既存の board + assistant drawer の再命名で終わらず、最初の実運用型として回ることを証明するため
