## ユーザーストーリー

**誰が**: 配下の AI 作業者群を 24/7 で回しつつ、境界条件と承認ゲートを握る現場責任者（佐藤 or 職能リード）  
**何を**: Brainbase 的な作業面を詳細作業面に下げた上で、施策選定、優先順位判断、作業者監督、短い介入対話、深い論点整理を 1 つの現場責任者コンソールで行いたい  
**なぜ**: 作業者のセッションやログを逐一見に行く運用では、現場責任者自身が作業者化してしまい、自律組織の管理者 / 統制役として機能しなくなるから

## 背景

- UnsonOS v0 の起点は経営者向け画面よりも現場責任者向け画面のほうが現実の運用と接続しやすい
- Brainbase はすでに作業台 / セッション / ファイル / ターミナルの作業面として強い
- ただし現在の Brainbase だけでは、現場責任者が配下作業者群の 24/7 稼働状態、リスク、承認、配分を管理者視点で扱いにくい
- 現場責任者は区画 / 建物の責任者であり、作業者の代替ではなく管理者 / 承認者 / 例外処理役として振る舞う必要がある
- 対話は必要だが、チャット主役に戻すと現場責任者向け UI ではなくアシスタント UI に逆流する

## 目指す状態

- 現場責任者が `注力テーマ・施策一覧`、`優先順位・配分表`、`判断待ち一覧`、`進行レーン`、`稼働状況パネル` を主画面として使える
- 軽い介入は `即応パネル` で行い、対象オブジェクトに結びついた短い対話と操作へ接続できる
- 深い壁打ちは `論点整理室` に逃がし、事業責任者へ上げる前の論点整理、反対意見、リスク確認を、会話記録を下層保持しつつ判断支援オブジェクトに変換できる
- 同じ判断が繰り返される場合、AI が `裁量昇格候補` を提案し、人間が条件付きの事前許可へ昇格できる
- 作業者の生セッションや操作台は `作業台` に閉じ、主画面の主語を壊さない

## 目指す体験

現場責任者としての理想体験は次の通り。

- 朝に全文を読むのではなく、**今日裁くべきものだけが最初に見える**
- 配下の AI が 24/7 で動いていても、**危ない場所だけ先に分かる**
- 何かが止まった時、**まず原因と選択肢が短く出る**
- 判断したら、その判断が **再割り振り / 成果物要求 / 上申** にすぐつながる
- 毎回同じ確認が続くなら、**次回からの事前許可候補** が見え、聞き直しを減らせる
- 深い壁打ちが必要な時だけ `論点整理室` に入り、終わったら **上申資料 / 判断メモ / 判断理由** に変換される
- 1 日の終わりに、**自分が何を裁いたか** と **何が残っているか** が分かる

この体験の画面分担は次で固定する。

- **左で、現場が生きている**
- **中央で、選んだ対象の進行を読む**
- **右で、人間が裁いている**
- **下で、今この場面に関係する役付きAIが前に出る**

この時、主ナビは常に `場所か異常を選ぶ -> その後で担当AIが前に出る` である。
人物選択を主ナビにしないことで、chat-first への逆流を防ぐ。

## 代表的な 1 日の流れ

### 8:45 出勤直後

- `区画ヘッダー` で `停滞中 / 承認待ち / 成果物不足 / 前日からの変化` を 3 秒で掴む
- `判断待ち一覧` と `稼働状況パネル` を見て、今日先に裁くべき施策を絞る
- この段階では生ログや会話記録を読まない
- 体験の要点は、**まず全体を読むのではなく、まず裁く対象が見える** こと

### 9:00 朝一の確認

- `稼働状況パネル` では、AI 作業者群の全文会話ではなく
  - 停滞
  - 承認待ち
  - 成果物不足
  - 実行失敗
  - 再試行
  の変化だけを短く把握する
- 現場責任者は `AI が夜の間も動いていたこと` を感じるが、ログ監視要員にはならない
- 体験の要点は、**AI の 24/7 稼働が気配ではなく異常の種類として見える** こと

### 9:15 1 件目の介入

- 停滞中の施策を開く
- `即応パネル` で
  - 状況要約
  - 割り振り支援
  - リスク確認
  - 証跡確認
  を短く回す
- 返ってくるのは長文会話ではなく
  - 3 行要約
  - 主因 / 副因
  - 選択肢
  - 必要承認
  - 足りない成果物
  である
- その場で
  - 再割り振り
  - 成果物要求
  - 保留
  を行う
- 同じ確認が続いている場合は `裁量昇格候補` を見て、次回からの事前許可候補として裁ける
- 体験の要点は、**短い壁打ちの出口がすぐ操作につながる** こと

### 10:00 作業者への反映

- 現場責任者は prompt や tool call を手で書き換えるのではなく、
  - 何を揃えるか
  - どの AI を何枠増やすか
  - どの対象を保留にするか
  だけを決める
- 指示は `対象 / 要求 / 期限 / 必要証跡 / 優先度` の形で生成され、AI 作業者や必要な人間作業者へ渡る
- Slack などへ通知は飛んでよいが、正本は UnsonOS に残る
- 体験の要点は、**自分は何を揃えるかだけ決めればよく、細かい依頼文の作成に落ちない** こと

### 11:30 2 件目の介入

- 次は `リスク確認` が必要な施策を開く
- `リスク確認` は
  - どこが危ないか
  - 何を限定すれば進められるか
  - 上位承認が必要か
  を返す
- 現場責任者は `全面展開 / 限定実施 / 保留` のどれで行くかを裁き、`判断理由` を残す
- 体験の要点は、**AI が代わりに決めるのではなく、裁きやすい粒度まで圧縮してくれる** こと

### 13:00 見回り

- 区画全体を見て
  - 緑 = 順調
  - 黄 = 要注意
  - 赤 = 介入要
  を把握する
- 色は雰囲気ではなく
  - 停滞
  - 承認待ち
  - 成果物不足
  - 実行失敗
  - 上申候補
  に基づいている
- 体験の要点は、**現場の空気が分かるが、雰囲気だけで終わらない** こと

### 14:30 想定外の異常

- 実行失敗が複数回続く施策では、`即応パネル` だけでは足りないと判断する
- この時点では
  - 原因候補が複数
  - 他施策への影響あり
  - 停止 / 縮小 / 継続の比較が必要
  という形で見える
- 体験の要点は、**例外案件だけ深い整理へ進む** こと

### 14:40 論点整理室

- `論点整理室` は blank なチャットではない
- 最初から
  - 事業
  - 区画
  - 施策
  - 現在状態
  - 影響範囲
  - 判断履歴
  - 足りない証跡
  が読み込まれている
- 現場責任者は
  - 進める案 / 縮小案 / 停止案
  - 反対意見
  - 事業責任者へ説明するための要点
  を深く壁打ちする
- 右側には
  - 論点
  - リスク
  - 足りない証跡
  - 候補アクション
  - 上申要点
  - 判断理由案
  が構造化されて積み上がる
- 体験の要点は、**考えること自体がそのまま上申資料の素材になる** こと

### 15:15 上申

- `上申資料を作成` を押すと、事業責任者の `判断待ち一覧` にそのまま上がる
- 現場責任者は、別フォーマットに書き直す必要がない
- 体験の要点は、**上に上げるたびに二重作業しなくて済む** こと

### 16:00 外部ツールとの併用

- Slack や会議は残る
- ただし、判断に影響するやり取りは対象 object に戻される
- Slack スレッドそのものを正本にしない
- 体験の要点は、**既存ツールを捨てなくてよいが、組織の記憶が散らばらない** こと

### 17:00 夕方の締め

- 同じ区画画面で
  - 今日裁いた判断
  - 今日出した成果物要求
  - 今日変えた割り振り
  - 上申中の件数
  - 明日に持ち越す停滞
  を確認する
- 現場責任者が「今日も Slack とログに溺れた」ではなく、**管理者として裁けた** 感覚で終われることを重視する

## シーン別に成立しているべきこと

### シーン 1: 朝の把握

- 全文ログではなく、今日裁くべき項目が先に見える
- 優先順位が最初から絞られている

### シーン 2: 軽い介入

- 会話のための会話にならない
- 判断と操作が直接つながる

### シーン 3: 深い整理

- 深い壁打ちが無駄にならず、上申資料や判断メモへ変わる
- 同じ論点の書き直しが減る

### シーン 4: 外部ツールとの併用

- Slack やメールを残しても、正本は散らばらない

### シーン 5: 1 日の締め

- 作業量ではなく、管理者として裁いた判断の数と質が分かる

## 受け入れ条件

| AC | 内容 | 検証方法 |
|----|------|---------|
| AC1 | 現場責任者の主画面が作業者セッション一覧ではなく、施策 / 配分 / 進行 / 判断を中心に構成される | ストーリーを読んで主画面の主語が管理者視点に固定されていることを確認 |
| AC2 | 現場責任者が AI 作業者群の活動を会話記録ではなく `状態 / 変化量 / 連鎖の健全性` として監督できる前提が含まれる | 目指す状態と背景に 24/7 活動可視化の粒度が書かれていることを確認 |
| AC3 | 軽い対話と深い壁打ちが分離され、`即応パネル` と `論点整理室` の二段構えになっている | ストーリーを読んで対話面の役割分離が明確であることを確認 |
| AC4 | `論点整理室` が `判断待ち一覧 / 進行 / 監査・証跡` から開く、対象付きの論点整理面として定義されている | 深い壁打ちが空のチャットではないことを確認 |
| AC5 | AI 返答の終点が会話記録そのものではなく `操作` または `判断メモ / 上申資料 / 判断理由` に接続される | 壁打ちの主成果物が会話ではなく判断オブジェクトとして扱われることを確認 |
| AC6 | Brainbase の作業台が必要な作業面として残る一方、現場責任者コンソールの主画面には逆流しない方針が明記される | Brainbase の位置づけが詳細 / 作業台側に固定されていることを確認 |
| AC7 | 現場責任者の 1 日の流れとして `朝の把握 -> 軽い介入 -> 深い整理 -> 夕方の締め` が成立する | ストーリーを読んで、画面部品が時系列の体験としてつながっていることを確認 |
| AC8 | `稼働状況パネル` が AI の全文会話ではなく、変化と異常の圧縮表示を返す前提になっている | 24/7 稼働の可視化がログ監視ではないことを確認 |
| AC9 | `論点整理室` の価値が、深い会話そのものではなく上申資料や判断メモへ変換されることだと明記される | 深掘りが成果物化することを確認 |
| AC10 | 同じ承認や再割り振りが繰り返される場合、AI が `裁量昇格候補` を出し、人間が条件付きで事前許可へ昇格できる | ask-first を減らす仕組みがストーリーに含まれていることを確認 |
| AC11 | AI は裁量候補を提案できても、自分で権限を自己承認できない | 裁量学習と権限自己拡張が分離されていることを確認 |
| AC12 | `朝の把握 -> 軽い介入 -> 作業者への反映 -> 深い整理 -> 上申 -> 夕方の締め` の 1 日の体験が具体的なシーンとして読める | ストーリーを読んで、現場責任者の時間軸に沿って UI が説明されていることを確認 |
| AC13 | 外部ツールとの併用が残っても、判断に効く情報は対象 object に戻される | Slack や会議の結果が UnsonOS の正本へ戻る前提があることを確認 |

## 優先度

**P0** - Brainbase を UnsonOS へ進化させる最初の実運用面であり、管理者と作業者の境界を保つ起点だから
