# Frame: Mesh — AI Agent間リアルタイム連携による経営の脱・属人化

## この世界では何が真か

一人社長＋業務委託メンバー（3〜7名）の小規模チームにおいて、社長の脳が唯一の「全体像統合ポイント」である。各委託メンバーへの状況確認・判断・調整のすべてが社長を経由し、社長が寝ている間は何も進まない。

この構造は、AI駆動経営フレームワーク（セクション1）の「人がボトルネックになる3つの原因」がそのまま当てはまる：
- **情報が散らばる** — 各メンバーのPC上のAI Agentはそれぞれ豊富な作業文脈を蓄積しているが、Agent間で繋がっていない
- **判断を人に依存** — 委託メンバー同士の連携は常に社長を経由する
- **プロセスが旧態依然** — Slackで聞いて、返事を待って、脳内統合という人力プロセス

## 前提

- 各メンバーは既にClaude CodeなどのAI Agentを日常的に使っている
- タスク管理はNocoDB（中央DB）で共有済み
- Slackでチーム全員が繋がっている
- 各メンバーのPC上のAI Agentは、数日〜数週間分の作業文脈（コード状態、試行錯誤、設計意図）を蓄積している
- しかしAgent間に直接通信する手段がなく、この文脈は各PC内に閉じている

## 設計思想

### 価値の核心は「データ共有」ではなく「Agent間リアルタイム問い合わせ」

ファイル共有ならGoogle Drive、タスク共有ならNocoDBで既にできている。Meshの存在意義は「各ノードのAIがローカル文脈を保持したまま、他ノードからの質問に回答を生成して返せること」にある。生データは一切送信しない。

### 3つの価値

1. **社長の脳のアンロード** — 社長のAIが全ノードに一斉問い合わせ → 数秒で全体像統合 → 社長は判断だけに集中
2. **委託間の直接連携** — 委託AのAIが委託BのAIに直接質問できる（社長を起こさなくていい）
3. **暗黙知の即座の引き出し** — 各Agent蓄積した作業文脈に基づいた回答が、質問に応じて即座に返る

### 鍵 = 問い合わせ範囲 = 組織権限

CEO鍵は全ノードに問い合わせ可能。Worker鍵は同プロジェクトのノードのみ。暗号の鍵配布が組織の権限設計そのものになる。

### brainbase 5原理との整合

| 原理 | Meshでの適用 |
|------|-------------|
| A: 正本主義 (SSOT) | メタデータのSSOTはNocoDB（変更なし）。各ノードのローカル文脈はそのノードが正本 |
| B: フロー/スタック分離 | Query/Responseはフロー（残さない）。NocoDB・Wikiはスタック（残す） |
| C: 状態管理主義 | NoCoDB上のタスク状態が会社の計器。Meshは計器を高速に読む手段 |
| D: 責任の一意化 | ROLE_RANKがQuery権限を定義。誰が何を聞けるかが一意 |
| E: 自律運転への収束 | Agent間が自律的に連携し、社長は例外承認のみ |

## 語彙

| 用語 | 定義 |
|------|------|
| **Node** | チームメンバーのPC上で動くBrainbaseインスタンス |
| **Mesh** | 複数のNodeがRelayを介して形成する通信ネットワーク |
| **Relay** | 暗号化されたenvelopeを転送する中継サーバー。内容は復号できない |
| **Query** | あるNodeのAIから別のNodeのAIへの問い合わせ |
| **QueryHandler** | Queryを受けたNodeがローカル文脈を収集して応答を生成する仕組み |
| **Envelope** | 暗号化された通信単位。from, to, type, payload, 署名を含む |
| **ROLE_RANK** | 組織上の役割レベル。CEO=3, GM=2, Member=1。Query権限とSecret復元範囲を決定 |

## スコープ

このFrameに属するStory群：
- Mesh基盤構築（Agent間通信の確立）
- Query層の実装（Agent間リアルタイム問い合わせ）
- Secret層の実装（CEOのみ復元可能な機密データ保護）
- 本体統合（既存Brainbase server.jsへのMesh統合）
- Slackログイン統合（メッシュ参加の自動化）
- CEO統合ビュー（全ノード状態ダッシュボード）

## スコープ外

- 分散推論（OSSモデル＋コミュニティPCリソース共有）→ 将来のFrame
- WebRTCフルP2P → Mesh基盤の将来改善
- モバイル対応 → 別Frame

## 判断軸

「この機能いる？」の判定：
1. それは**社長が各委託に個別に聞く回数を減らす**か？
2. それは**委託メンバー同士が社長を介さずに連携できる**ようにするか？
3. それは**生データを送信せず、AIが生成した回答だけを流す**設計か？
4. それは**既存のSlackログイン＋NoCoDBの仕組みを壊さない**か？
