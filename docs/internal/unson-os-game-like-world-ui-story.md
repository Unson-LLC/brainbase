## ユーザーストーリー

**誰が**: AI 組織を抱えた複数事業を運営する経営者兼実務担当（佐藤）  
**何を**: 単なる管理画面ではなく、事業都市の全体像を俯瞰しながら異常や停滞を直感的に掴める UI で運用したい  
**なぜ**: チャットや表中心の画面では、組織の空気、詰まり、成長、危険の分布を一目で把握しづらいから

## 背景

- UnsonOS の UI はタスクボードやチャットパネルの延長ではなく、経営インターフェースとして設計する必要がある
- 世界観は `事業拠点 / 事業都市` を上位に据え、局所では実験室 / スタジオ / 現場フロア / 証跡庫 / 指令室 を使い分ける
- ただし運用ラベル自体は実務語のまま保ち、メタファーは構造と空間の翻訳に使う
- ゲームっぽさは装飾ではなく、複雑な状態を直感理解させるための認知インターフェースとして扱う

## 目指す状態

- 平常時は都市や区画の状態を見るだけで、どこが元気でどこが停滞しているか分かる
- 異常時は全体俯瞰からそのまま進行 / 承認 / 監査に降りられる
- 事業ごとの性格差、探索フェーズ、定常運用フェーズが同じ UI 世界の中で共存できる

## 受け入れ条件

| AC | 内容 | 検証方法 |
|----|------|---------|
| AC1 | 上位世界観が `事業拠点 / 事業都市` として固定されている | ストーリーを読んで最上位の世界観が明確であることを確認 |
| AC2 | ゲームっぽさの目的が装飾ではなく認知負荷の低減だと明記される | 背景を読んで UI の意図が明確であることを確認 |
| AC3 | `全体俯瞰面 -> 経営判断面 -> 進行管理面 -> 監査面` への導線が想定されている | 目指す状態と受け入れ条件に 4 層の接続が読み取れることを確認 |
| AC4 | `停滞中 / 承認待ち / 実行失敗 / 成果物不足` などの実務ラベルを維持する方針が含まれる | 背景に運用ラベル維持方針があることを確認 |
| AC5 | 事業ごとの異質さを区画や建物の性格差で扱う前提が含まれている | 目指す状態に事業固有性が含まれていることを確認 |
| AC6 | 後続アーキテクチャで区画 / 建物 / 部屋 / レーン などの空間モデルへ落とせる | ストーリーからワールドモデルの抽象境界が読み取れることを確認 |
