## ユーザーストーリー

**誰が**: AI 組織を自律的に動かしたいが、意味変更や損害範囲は制御したい経営者兼承認者（佐藤）  
**何を**: ストーリーから実行までを DAG / チェックポイント / 承認ゲートで制御し、低リスクは流しつつ高リスクだけ止めたい  
**なぜ**: skills や prompt だけでは進行順序や停止条件を強制できず、放置可能な self-driving 運転が危険になるから

## 背景

- OpenGoat の観察から、タスク分解や自動割り振りには価値がある一方、完走率と制御性はまだ弱いことが見えている
- UnsonOS は `ストーリー駆動パイプライン` を本体に持ち、`ストーリー -> スコープ -> 計画 -> 実行 -> 検証 -> 承認 -> 出荷 -> 学習` の進行を制御する必要がある
- ガードレールの判断軸は少なくとも `意味差分` と `損害範囲` を含む
- 人間は全実行を読むのではなく、ゲートをまたぐ時と高リスク例外時だけ介入する

## 目指す状態

- AI 組織が自律実行しても、どこで止まるか、何を根拠に進むかが明確になる
- 承認が必要な変更だけが人間に上がり、低リスク作業は unattended に流れる
- ストーリーから成果物、証跡、学習まで 1 本の進行線として追跡できる

## 受け入れ条件

| AC | 内容 | 検証方法 |
|----|------|---------|
| AC1 | UnsonOS の制御主体が prompt ではなく DAG / チェックポイント / ゲートであると明記される | 背景を読んで制御主体が明確であることを確認 |
| AC2 | `ストーリー -> スコープ -> 計画 -> 実行 -> 検証 -> 承認 -> 出荷 -> 学習` の最小進行線が定義されている | ストーリーに進行段階が揃っていることを確認 |
| AC3 | low-risk 自動実行と high-risk 承認の分岐前提が含まれている | Desired Outcome と AC に unattended と approval の両立があることを確認 |
| AC4 | ガードレールの判断軸として `意味差分` と `損害範囲` が含まれている | 背景に評価軸が明記されていることを確認 |
| AC5 | 人間の役割が全件レビューではなくゲート承認者へ上がる | ユーザーストーリーと目指す状態に人間の役割上昇が明記されていることを確認 |
| AC6 | 後続アーキテクチャ / 仕様で policy engine, checkpoint model, evidence ledger へ接続できる | ストーリーから必要な概念境界が読み取れることを確認 |
