## ユーザーストーリー

**誰が**: 複数事業を同時に持ち、AI 組織へ資源配分する経営者（佐藤）  
**何を**: 事業横断の状態、投資先、承認待ち、危険な滞留を 1 つの上位画面で把握して判断したい  
**なぜ**: session や task を個別に追うだけでは、どの事業に張るべきか、どこで止めるべきかを経営判断できないから

## 背景

- UnsonOS の主語は agent ではなく `事業群 / 事業 / 資源配分` である
- Brainbase は実務担当向け操作面として個別作業や詳細確認には強い
- OpenGoat は org runtime の参考実装として、AI 組織の自律実行感を示している
- ただし複数事業を横断して「どこに張るか」を判断する事業群層はまだ正面から扱えていない

## 目指す状態

- 経営者が複数事業の健全性、停滞、承認待ち、投資先を短時間で判断できる
- 各事業の内部詳細へ降りる前に、事業群全体の優先順位を調整できる
- 人間は細かな逐次管理ではなく配分 / 介入 / 承認に集中できる

## 受け入れ条件

| AC | 内容 | 検証方法 |
|----|------|---------|
| AC1 | 最上位 UI の主語が `セッション` や `agent` ではなく `事業 / 注力テーマ / 資源配分` になる | ストーリーを読んで最上位判断対象が事業群になっていることを確認 |
| AC2 | 経営者が平常時に見る情報と異常時に深掘る情報が分離されている | Desired Outcome と AC を読んで `俯瞰 -> 深掘り` の導線が明確であることを確認 |
| AC3 | 事業ごとの状態差を比較できる前提が含まれている | Context と Outcome に `事業横断比較` が明記されていることを確認 |
| AC4 | 実務担当向け操作面と事業群判断面の責務が混同されていない | Brainbase の役割が詳細操作面であり、最上位画面の主役でないことを確認 |
| AC5 | 人間の役割が細かな逐次管理ではなく配分 / 承認 / 上申であることが明示される | ユーザーストーリーと目指す状態を読んで人間の立場が経営者寄りで固定されていることを確認 |
| AC6 | 後続アーキテクチャで区画 / 事業 / リスク / 承認待ち一覧を扱う前提が接続できる | ストーリーから事業群層の概念境界が読み取れることを確認 |
