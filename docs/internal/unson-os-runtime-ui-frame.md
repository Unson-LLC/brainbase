## 目的

このフレームは、`Brainbase -> OpenGoat -> UnsonOS` の進化をぶらさずに進めるための上位前提を定義する。

- ストーリーより上位で扱う
- 仕様や実装詳細は書かない
- 以後のストーリー / アーキテクチャ / 仕様の判断基準を固定する

## 位置づけ

### 進化線

| レイヤ | 役割 | このフレームでの位置づけ |
|--------|------|------------------------|
| **Brainbase** | 実務担当向け操作面 | 人間が可視化・介入・承認するための司令塔 |
| **OpenGoat** | 組織実行基盤の参考実装 | AI 組織が自律実行する実行基盤の参考実装 |
| **UnsonOS** | 複数事業経営 OS | 複数事業を AI 組織込みで経営する上位 OS |

### 一言でいうと

- Brainbase は **操縦席**
- OpenGoat は **実行心臓**
- UnsonOS は **経営 OS 本体**

## 中核前提

### 1. UnsonOS の主語は agent ではなく事業

UnsonOS は単なる agent 管理ツールではない。  
主語は最終的に `事業群 / 事業 / 資源配分 / 承認 / 証跡` である。

### 2. AI はツールではなく組織ユニット

AI は単発のチャット相手ではなく、役割を持って継続稼働する組織ユニットとして扱う。  
人間は作業者ではなく、管理職・経営者・承認者として振る舞う。

### 3. 自律実行は必須だが、無制御自動化は採用しない

価値は self-driving にある。  
ただし、勝手に回ること自体が目的ではなく、`ガードレール / 承認 / 証跡` の上で回ることが必要。

### 4. Skills は助言、DAG は制御

skills や prompt は有効だが、強制力のある制御ではない。  
UnsonOS の本体は、`ストーリー駆動パイプライン` と `DAG / チェックポイント / ゲート` による強制的な進行制御にある。

### 5. UI は管理画面ではなく経営インターフェース

UI はタスクボードの延長ではない。  
人間が AI 組織と複数事業を経営するためのインターフェースとして設計する。

## 戦略軸

このフレームでは、UnsonOS を次の 4 軸で捉える。

### A. 事業群軸

- どの事業に資源を張るか
- どの注力テーマを継続 / 縮小 / 停止するか
- どの事業が利益と学習を生んでいるか

### B. 事業軸

- 各事業の事業責任者と職能リード
- その事業の目標、KPI、停滞要因
- 事業ごとの運営リズム

### C. 実行基盤軸

- role / agent / task / run / approval
- 委譲と実行
- heartbeat / loop / 自動割り振り

### D. ガバナンス軸

- 意味差分
- 損害範囲
- 承認ゲート
- 証跡確認
- 監査履歴

## 世界観

### 上位世界観: 事業拠点 / 事業都市

UnsonOS の上位世界観は **群島交易連邦** ではなく、**複数事業を抱える事業拠点 / 事業都市** とする。

理由:

- 事業ごとの差異を区画や建物として表現しやすい
- `事業群 / 事業 / 実行基盤 / ガバナンス` を 1 つの世界で共存させやすい
- 進行優先に寄りすぎず、探索・運営・監査を同じ世界に置ける
- 現実の業務用語を維持したまま、構造だけゲームライクにできる

### 局所メタファー

上位世界観は 1 つにしつつ、局所では仕事の性質に応じてメタファーを使い分ける。

| 場 | 向く仕事 |
|----|----------|
| **実験室 / スタジオ** | 探索、仮説、企画、ブランド、実験 |
| **現場フロア / 工房セル** | 実行、検証、出荷、反復改善 |
| **会議室 / ゲート** | 承認、着手判断、例外判断 |
| **証跡庫 / 保管庫** | 証跡、監査、成果物、追跡記録 |
| **指令室** | 事故対応、経路変更、上申、損害抑制 |
| **港 / 交易路** | 事業間・外部連携の進行表現 |

### メタファールール

メタファーは **構造に使うが、言葉までは翻訳しない**。

つまり UI 上では

- `停滞中`
- `承認待ち`
- `実行失敗`
- `成果物不足`

のような現実の運用語をそのまま使ってよい。  
ゲームっぽさは、配置・空間・動き・区画の性格で表現する。

## UI 原則

### 原則 1: チャットを主役にしない

チャットは詳細ビューであり、主画面ではない。  
主画面は `状態 / 進行 / 介入` を扱う。

### 原則 2: ゲームのように見えるが、中身は統制システム

UI はゲームのような空間性や存在感を持ってよい。  
ただし、見た目だけをゲーム化するのではなく、裏側の `DAG / 承認 / 証跡` を可視化する翻訳装置として機能することが必要。

### 原則 3: 世界観と制御を分離しない

`ゲームライクな世界` と `ワークフロー統制` を別物にしない。  
同じ状態を、

- 経営者にとっては「会社を見渡す世界」
- 実務管理にとっては「作戦線 / 生産ライン / チェックポイント」

として見せ分ける。

### 原則 4: 平常時は俯瞰、異常時だけ深掘り

人間が全ログを読む前提を捨てる。  
平常時は抽象化された全体俯瞰 / 経営判断 / 進行を見せ、異常時だけ生ログ / 成果物 / diff に降りる。

## UI ワールドモデル

UnsonOS の UI は次の 4 層で表現する。

### 1. 全体俯瞰面

事業拠点全体を俯瞰する層。

- 事業群の俯瞰
- 事業ごとの区画 / 建物
- 稼働・停滞・異常の気配
- 区画ごとの成熟度と役割差

### 2. 経営判断面

都市計画 / 投資配分として注力テーマと目標を扱う層。

- 今どこに張るか
- 何を止めるか
- どの注力テーマを伸ばすか

### 3. 進行管理面

DAG / パイプライン / チェックポイントを扱う層。  
ここだけは比喩を弱め、かなり実務語で表示してよい。

- ストーリー
- スコープ
- 計画
- 実行
- 検証
- 承認
- 出荷
- 学習

### 4. 監査面

詳細ログ、成果物、diff、証跡、承認履歴を扱う層。  
これは最も管理画面に近くてよい。

## メタファー構成

UnsonOS の UI は単一メタファーで固定しない。  
ただし、**上位世界観は事業拠点に固定**し、その内部で役割ごとに局所メタファーを使う。

### パーティ

- agent identity
- class / role
- 誰が何に強いか
- ただし agent を UI の主語にはしない

### オフィス

- 組織運営
- 部署
- 会議
- 承認
- 区画 / 建物 / フロアの責任構造

### 工場

- work item の流れ
- bottleneck
- QA / 出荷 / 再作業
- 実行レーン

### 戦略ゲーム

- 資源配分
- 事業群の統制
- チェックポイント
- 着手判断

### 港 / 交易路

- 事業間の進行
- 外部市場との接続
- channel / pipeline / distribution の可視化

## 人間の役割

人間の役割は次の順で上がっていく。

1. 実務担当
2. 管理者
3. 承認者
4. 事業群経営者

UnsonOS の UI は、この役割上昇を前提に設計する。  
人間が各 agent を逐一細かく管理しなくても済む形が基準である。

## 対象外

このフレームでは次をまだ決めない。

- API
- data schema
- implementation language
- exact screen layout
- concrete component tree

それらは後続のストーリー / アーキテクチャ / 仕様に落とす。
