## 位置づけ

この文書は、UnsonOS の UI を **見た目の世界観** ではなく **経営と運営の情報設計** として定義する。

- 見た目はゲームライクでよい
- ただしラベルは実務語を優先する
- 主画面はチャットではなく、判断と進行の面にする
- Brainbase は捨てず、詳細作業面として残す

UI と制御面の対応は、[UI×制御面 整合性マトリクス](./unson-os-ui-architecture-alignment.md) を参照する。

## UI の主語

UnsonOS の UI は次の順で主語を持つ。

1. 事業群
2. 事業
3. 注力テーマ / 施策
4. 進行
5. 監査・証跡
6. 詳細
7. 連絡と決定記録

画面表示では、以下を使う。

- `Portfolio` → 事業群
- `Business` → 事業
- `Bet` → 注力テーマ
- `Initiative` → 施策
- `Flow` → 進行
- `Audit` → 監査・証跡
- `Detail` → 詳細
- `Communication` → 連絡
- `Decision Record` → 決定記録

## レイヤー構成

### 1. 全体俯瞰面

経営者が最初に見る面。

- 複数事業の健全性
- 停滞の分布
- 承認待ちの偏り
- 資源配分の歪み
- 重大リスク

### 2. 経営判断面

何に張るか、何を止めるかを決める面。  
ここが board-first の主戦場。

主要構成:

- 注力テーマ・施策一覧
- 優先順位・配分表
- 判断待ち一覧
- 停滞一覧
- 証跡不足一覧
- 裁量昇格候補一覧

### 3. 進行管理面

施策がどこで進み、どこで止まり、どこで承認待ちになっているかを見る面。

主要構成:

- 進行レーン
- 稼働状況パネル
- 即応パネル

現場責任者にとっての理想体験は次の通り。

- 朝に全部を読むのではなく、今裁くべき項目が先に見える
- AI の 24/7 稼働は、全文会話ではなく異常の種類として見える
- 軽い介入は即応パネルで終わる
- 深い整理が必要な時だけ論点整理室へ降りる
- 深掘りの出口は transcript ではなく `上申資料 / 判断メモ / 判断理由` になる
- 同じ確認が繰り返されるなら、次回からの事前許可候補として `裁量昇格候補` を裁ける
- Slack や会議を残しても、判断に効く情報は対象 object と決定記録へ戻る

## 表ビューアの方針

UnsonOS は **汎用スプレッドシート** ではなく、**裁くための表ビューア** を持つ。

### UnsonOS に置く表

- 注力テーマ・施策一覧
- 優先順位・配分表
- 判断待ち一覧
- 停滞一覧
- 証跡不足一覧
- 上申一覧
- 承認履歴一覧

これらは、見るだけでなく、その場で裁く必要がある表である。

### UnsonOS の表ビューアが持つべき性質

- 行が `事業 / 区画 / 施策 / 判断 / 実行 / 成果物` の正体に結びつく
- 並び替え、絞り込み、グループ化、保存ビューがある
- 行を選ぶと、右側に即応パネルや論点整理室が開く
- 同じ判断が繰り返される場合は、`裁量昇格候補` を判断対象として扱える
- その場で `承認 / 差戻し / 上申 / 成果物要求 / 再割り振り` ができる
- 判断メモと証跡に直結する

### UnsonOS が最初から持たなくてよいもの

- 関数だらけの自由計算
- ピボットや自由集計のフル機能
- 大量セルの手入力補助
- 文書埋め込み前提の簡易テーブル
- 開発者向けの Issue 管理そのもの

### 4. 監査面

判断や承認の根拠を遡る面。

- 証跡
- 成果物
- 判断理由
- 上申履歴
- 会話記録の参照

### 5. 詳細作業面

Brainbase 側に寄せる面。  
raw session / chat / terminal / files はここに置く。

### 6. 連絡面

役割横断で、判断依頼、承認依頼、証跡要求、上申、差戻しを扱う共通面。  
主役は会話そのものではなく、**対象に紐づいた連絡 object** と **決定記録**。

主要構成:

- 受信箱
- 稼働通知
- 論点整理室
- 決定記録
- 裁量昇格候補

## 世界観

### 上位世界観

上位世界観は **事業圏 / 事業都市 / 区画 / 建物** とする。

ただし、これは正本ではなく投影である。

- 事業群 = 事業圏
- 事業 = 事業都市 / 事業拠点
- Project / Program = 区画
- Story / Run / Checkpoint = 建物 / 部屋 / レーン

### 重要な原則

- 世界観は装飾ではなく、縮尺と遷移の座標系である
- 画面ラベルは実務語を優先する
- `停滞中 / 承認待ち / 実行失敗 / 成果物不足` などはそのまま使う
- 世界観は連絡や判断の文脈を整えるが、正本そのものではない

## 画面名

表示上は次を使う。

- `World Layer` → 全体俯瞰面
- `Mission Layer` → 経営判断面
- `Flow Layer` → 進行管理面
- `Audit Layer` → 監査面
- `Detail Layer` → 詳細作業面
- `Communication Surface` → 連絡面

## 役割別の既定画面

### 経営者

- 既定画面: 全体俯瞰面 + 経営判断面
- 主操作: 配分、継続、停止、着手判断、下位への方針反映

### 事業責任者

- 既定画面: 経営判断面 + 進行管理面
- 主操作: 再優先順位付け、配分変更、承認、現場責任者からの上申処理

### 現場責任者

- 既定画面: 進行管理面
- 主操作: ゲート判断、再割り振り、成果物要求、上申

### 作業者

- 既定画面: 詳細作業面
- 主操作: 実作業、成果物返却、会話記録の参照

## 連絡面の原則

UnsonOS は **会話の場所** ではなく、**連絡の正本** を持つ。

### 連絡の目的分類

UnsonOS の連絡は、少なくとも次の 8 種類に整理する。

- 指示
- 報告
- 相談
- 判断依頼
- 承認依頼
- 証跡要求
- 差戻し
- 上申

### 連絡 object の最低限の項目

- 対象
- 発信者
- 宛先
- 目的
- 緊急度
- 期限
- 要求アクション
- 要約
- 参照証跡
- 配信先
- 結果

### AI⇆AI

AI⇆AI は、原則として人に全文を見せない。  
平常時は稼働状況パネルに圧縮表示し、異常時だけ「何が止まったか」「どこで跳ね返ったか」を見せる。  
深掘り時だけ詳細作業面でトレースを確認する。

### AI⇆人

AI⇆人 は、即応パネルと論点整理室の二段構えにする。  
入口は会話ではなく目的にする。

- 状況要約
- 停滞診断
- 割り振り支援
- リスク確認
- 証跡確認
- 上申整理

返答は長文ではなく、要約カード、原因カード、選択肢カード、リスクカード、証跡不足カード、上申資料案で返す。
同じ判断が繰り返される場合は、`この条件付き判断を、次回から事前許可扱いに昇格するか` を構造化した `裁量昇格候補` として返す。

### 人⇆人

人⇆人 も、雑談チャットを正本にしない。

- 対象に紐づく短い議論
- 会議
- 決定記録

の 3 つに整理し、重要な結論は必ず UnsonOS に戻す。

## 現場責任者向け対話

現場責任者向け対話は二段構えにする。

### 即応壁打ち

- 面: 即応パネル
- 特徴: 対象付き、短い往復、操作に直結
- 用途:
  - 状況要約
  - 割り振り支援
  - リスク確認
  - 証跡確認

### 深掘り壁打ち

- 面: 論点整理室
- 特徴: 深い論点整理、上申準備、判断理由の生成
- 入口: 判断待ち一覧 / 進行 / 監査面からのみ

### raw session / chat

- 面: 詳細作業面 / Brainbase
- 扱い: 保持はするが、主成果物にしない

## 壁打ちの成果物

残す成果物は transcript ではなく次とする。

- 判断メモ
- 上申資料
- 判断理由

会話記録は下層に保持し、事後検証や障害解析に使う。

## 既存ツールの位置づけ

### 残すが、正本にしないもの

- Slack: 通知、軽い承認、UnsonOS への入口
- Gmail: 社外連絡、正式送付、受信取り込み
- Notion: 方針文書、仕様メモ、プレイブック
- NocoDB: 受付フォーム、一時的な構造化データ置き場

### 強く残すもの

- GitHub: コード、PR、Issue、CI/CD、リリース証跡
- Drive: 成果物倉庫、証跡倉庫
- Calendar: 会議、承認枠、レビュー枠

### UnsonOS に寄せるもの

- 判断待ち一覧
- 上申資料
- 判断メモ
- 判断理由
- 証跡参照
- 稼働状況パネル
- 連絡台帳

### 捨てるべき重複運用

- Slack スレッドを判断履歴にする
- Notion DB と NocoDB の二重台帳
- Gmail を社内運用の主経路にする
- Drive フォルダで状態を表現する

## 表の役割分担

### UnsonOS に置く表

裁くための表を持つ。

- 経営判断表
- 進行管理表
- 監査・証跡表

### Google Sheets に残す表

作るための表を持つ。

- 一括入力
- 手修正
- 一時的な整形
- アドホック集計

### Notion に残す表

文書に付属する参照表を持つ。

- 仕様索引
- 用語集
- 会議ノート一覧
- プレイブック一覧

### GitHub に残す表

コードに近い表を持つ。

- Issue
- PR 連動の進行
- 開発ロードマップ

### NocoDB に残してよい表

移行期の補助台帳を持つ。

- 受付台帳
- フォーム入力の受け皿
- 短命な補助テーブル

## 命名ルール

- 画面名は日本語
- 状態名も日本語
- コード / DB / 外部連携キーだけ英語維持
- 世界観語は見出しや投影に使い、表の列名や判断項目は実務語で書く

## 移行方針

- 上位の世界観と判断面は UnsonOS 側で持つ
- 詳細作業面は Brainbase 側へ寄せる
- 既存の session / terminal / files をすぐ消さない
- 最初は現場責任者面を先に成立させ、その後に事業責任者 / 経営者面へ広げる
