# NocoDB 操作ガイド

NocoDBのデータ登録・更新・レポート生成に関する標準フォーマットとルール。

## 🚨🚨🚨 作業開始前チェック（絶対厳守）

**NocoDBへのデータ登録・更新作業を行う前に、必ず以下を確認すること。**

### 必須確認事項

1. **要件登録時**: 要求→要件ワークフロー（本ファイル538行目〜）を確認
   - 要求（BFD）を先に作成または特定
   - 要件（FRD）作成後、必ず `要求_id` で紐付け

2. **バグ登録時**: バグ登録ワークフロー（本ファイル655行目〜）を確認
   - 全必須フィールドを埋める
   - 原因ファイル・行番号を特定してから登録

3. **フィールド値の検証**: 登録前にオプション値を確認
   ```bash
   # テーブルのフィールド定義を確認
   ./tools/nocodb/cli.sh get 要件 1 | jq 'keys'
   ```

### 違反した場合の影響

- 要求と要件が紐付かず、トレーサビリティが失われる
- 不完全なデータが蓄積し、レポート品質が低下する
- 後から手動で紐付け作業が必要になる

## 🚨🚨🚨 最重要：カラムオプション更新時の注意（絶対厳守）

**⚠️ SingleSelect/MultiSelectカラムのオプション更新は既存データを破壊する可能性がある**

### 禁止事項

```bash
# ❌ 絶対禁止：新しいオプションで上書き
mcp__nocodb__nocodb_update_column with new options array
# → 既存レコードの値が全てnullになる
```

### 安全な更新手順

1. **事前バックアップ**（必須）

   ```bash
   ./tools/nocodb/cli.sh list {テーブル} 500 > /tmp/backup_YYYYMMDD.json
   ```

2. **Airtableとの同期確認**
   - Airtableに同じデータがあることを確認
   - 復旧用データソースとして使用可能

3. **オプション追加のみ**（推奨）
   - NocoDB UIから手動で絵文字・色を変更
   - 既存オプションのタイトル変更は安全

### 復旧手順（データが消えた場合）

```bash
# 1. Airtableからデータ取得
mcp__airtable__list_records

# 2. マッピング付きで復旧スクリプト生成
python3 generate_restore_script.py

# 3. バッチ更新実行
zsh /tmp/restore_status.sh
```

## 🚨 必須ルール

### 0. 工数の定義

**1人日 = 7時間**（8時間労働 - 休憩1時間）

| 単位    | 時間    |
| ------- | ------- |
| 0.5人日 | 3.5時間 |
| 1人日   | 7時間   |
| 2人日   | 14時間  |
| 3人日   | 21時間  |
| 5人日   | 35時間  |

**工数見積もりの優先順位**:

1. NocoDBの「見積もり（人日）」フィールドに値があればそれを使用
2. 値がなければ以下の基準で推定：
   - 単一ファイル・UIのみ → 0.5人日
   - 複数ファイル・ロジック追加 → 1〜2人日
   - システム全体・新機能 → 3〜5人日

docs/reports/2026-01-12-requirements-priority-report.md
docs/reports/2026-01-12-bugs-priority-report.md

## 使用例

ユーザーが「直近対応が必要な要件を10個ピックアップしてレポートにまとめて」と依頼した場合:

1. NocoDBからデータ取得
2. 優先度・影響範囲・ビジネス価値で分析
3. 上記フォーマットでレポート生成
4. **全てのIDにNocoDBへのURLリンクを付与**
5. **`docs/reports/YYYY-MM-DD-{種別}.md` にファイル保存**

## バグステータス管理（A案）

### ステータスフロー

```
🔍 新規報告 → 📋 確認中 → 🔧 修正中 → 📝 PR作成済み → 🧪 Preview検証中 → ✅ Preview検証済み → 🚀 本番反映済み → ✔️ 本番検証済み → 🏁 クローズ
```

### ステータス定義とカラー設定

| ステータス      | 絵文字 | 背景色HEX | 説明                                                             |
| --------------- | ------ | --------- | ---------------------------------------------------------------- |
| 新規報告        | 🔍     | `#ffc9c9` | バグが新規に報告された状態。まだ誰も確認していない               |
| 確認中          | 📋     | `#ffd8a8` | 開発者が内容を確認・調査中。再現手順の検証や原因特定を行っている |
| 修正中          | 🔧     | `#fff3bf` | 修正作業に着手。コードを書いている最中                           |
| PR作成済み      | 📝     | `#a5d8ff` | PRを作成しレビュー待ち。まだマージされていない                   |
| Preview検証中   | 🧪     | `#d0bfff` | Preview環境にデプロイ済み。開発者がテスト中                      |
| Preview検証済み | ✅     | `#b2f2bb` | Preview環境でのテスト完了。本番デプロイ待ち                      |
| 本番反映済み    | 🚀     | `#99e9f2` | 本番環境にデプロイ済み。本番での動作確認待ち                     |
| 本番検証済み    | ✔️     | `#69db7c` | 本番環境での動作確認完了。クローズ待ち                           |
| クローズ        | 🏁     | `#dee2e6` | 対応完了。このバグは解決済み                                     |

### ステータス遷移の判断基準

| 現在               | 次へ               | トリガー・条件                          |
| ------------------ | ------------------ | --------------------------------------- |
| 🔍 新規報告        | 📋 確認中          | 開発者がバグを認識し、調査を開始した時  |
| 📋 確認中          | 🔧 修正中          | 原因が特定され、修正作業に着手した時    |
| 🔧 修正中          | 📝 PR作成済み      | PRを作成しレビュー依頼した時            |
| 📝 PR作成済み      | 🧪 Preview検証中   | PRがマージされPreview環境に反映された時 |
| 🧪 Preview検証中   | ✅ Preview検証済み | Preview環境でのテストが完了した時       |
| ✅ Preview検証済み | 🚀 本番反映済み    | 本番環境にデプロイされた時              |
| 🚀 本番反映済み    | ✔️ 本番検証済み    | 本番環境での動作確認が完了した時        |
| ✔️ 本番検証済み    | 🏁 クローズ        | 報告者または管理者が完了を承認した時    |

### カラー設計の意図

```
進捗フロー（色の流れ）:
赤 → オレンジ → 黄 → 青 → 紫 → 薄緑 → シアン → 濃緑 → グレー

暖色系（未着手〜作業中） → 寒色系（レビュー〜検証） → 緑系（完了）→ グレー（終了）
```

## バグ重要度管理

### 重要度定義とカラー設定

| 重要度   | 絵文字 | 背景色HEX | 説明                                                   |
| -------- | ------ | --------- | ------------------------------------------------------ |
| Critical | 🔴     | `#fa5252` | 最優先対応。システム停止・データ損失・セキュリティ問題 |
| High     | 🟠     | `#fd7e14` | 早急対応。主要機能の障害・業務がブロックされる         |
| Medium   | 🟡     | `#fab005` | 計画的対応。機能制限はあるが回避策がある               |
| Low      | 🟢     | `#40c057` | 余裕があれば対応。軽微な問題・UX改善レベル             |

### 重要度の判断基準

| 重要度      | 判断基準                                                                                                                                                           |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 🔴 Critical | ・本番環境でサービスが停止している<br>・データ損失の可能性がある<br>・セキュリティ脆弱性が発見された<br>・売上に直接影響している<br>・全ユーザーが影響を受けている |
| 🟠 High     | ・主要機能が使用できない<br>・多数のユーザーに影響がある<br>・回避策がない<br>・顧客からクレームが発生している<br>・SLAに抵触する可能性がある                      |
| 🟡 Medium   | ・機能制限はあるが回避策が存在する<br>・一部のユーザーのみに影響<br>・UXが著しく低下している<br>・パフォーマンスに問題がある<br>・次回リリースまでに対応すべき     |
| 🟢 Low      | ・見た目・表示の問題<br>・軽微なUX改善<br>・エッジケースでのみ発生<br>・改善要望レベル<br>・対応しなくても業務に支障なし                                           |

## 要求→要件ワークフロー（必須）

### 🚨 基本ルール

**要求（BFD-xxx）を登録・確認した場合、必ず対応する要件（FRD-xxx）を作成すること。**

### ワークフロー

```
要求登録 → 要件作成 → 開発着手
  (BFD)     (FRD)
```

1. **要求登録時**: 要求テーブルにBFD-xxxが登録される
2. **要件作成**: 対応するFRD-xxxを要件テーブルに作成
3. **紐付け**: 関連バックログフィールドにBFD-xxxを記載

### 要件作成コマンド

**Step 1: 要件レコード作成**

```bash
./tools/nocodb/cli.sh create 要件 '{
  "要件名": "{要求内容を具体的な実装タスクに変換}",
  "機能ドメイン": "{適切なドメインを選択}",
  "開発ステータス": "📋 未着手",
  "詳細説明": "【概要】\n{概要}\n\n【対応範囲】\n- {項目1}\n- {項目2}",
  "見積もり（人日）": {数値},
  "受け入れ基準": "- [ ] {基準1}\n- [ ] {基準2}",
  "影響範囲": "{高/中/低 - 説明}",
  "ビジネス価値": "{適切な価値を選択}",
  "関連バックログ": "BFD-{番号}",
  "タグ": "{カンマ区切りタグ}"
}'
# 出力: {"番号": 462} ← この番号をStep 2で使用
```

**Step 2: 要求へのリンク設定（必須）**

```bash
./tools/nocodb/cli.sh update 要件 {作成された番号} '{"要求_id": {要求の番号}}'
# 例: ./tools/nocodb/cli.sh update 要件 462 '{"要求_id": 133}'
```

⚠️ **重要**: Step 2を忘れると要求と要件が紐付かないので必ず実行すること

   - 不確実性が高い場合は高めに見積もる
