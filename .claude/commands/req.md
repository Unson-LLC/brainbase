# 要件ベース開発（REQ-XXX）

新機能を要件番号で追跡しながら開発します。SalesTailorの要件追跡システムを適用。

## 引数
$ARGUMENTS: 要件ID（例: REQ-001）と概要

## 実行手順

1. **要件IDの確認・採番**
   - 既存の要件ID: `git log --oneline | grep -oE "REQ-[0-9]+" | sort -u | tail -5`
   - 新規の場合は次の番号を採番

2. **作業ブランチ作成**
   ```bash
   git checkout -b feature/REQ-XXX-<機能名>
   ```

3. **実装 → こまめにコミット**
   ```
   feat(scope): <機能概要>（REQ-XXX）

   目的:
   - この機能が解決する課題

   実装内容:
   - 主な変更点
   ```

4. **完了時のコミット**
   ```
   feat: REQ-XXX <機能名>の実装完了

   実装内容:
   - 機能A
   - 機能B

   REQ-XXX
   ```

## SalesTailor流のコミット例

```
feat: 企業選択時の送信履歴表示機能の実装（REQ-057 Phase 1）

目的:
- 企業を選択した際に過去の送信履歴を確認できるようにする
- 重複送信を防止する

実装内容:
- 送信履歴APIの追加
- 企業詳細モーダルに履歴タブ追加
- フィルター機能の実装

REQ-057
```

## 命名規則

| 種類 | プレフィックス | 例 |
|------|---------------|-----|
| 要件 | REQ-XXX | REQ-001, REQ-057 |
| バグ | BUG-XXX | BUG-001, BUG-036 |
| タスク | TASK-XXX | TASK-005 |
| フロント要件 | FRD-XXX | FRD-366 |
| ユーザーストーリー | US-XXX | US-050 |

## チェックリスト

- [ ] 要件IDが一意であること
- [ ] ブランチ名に要件IDが含まれていること
- [ ] 全コミットに要件IDが含まれていること
- [ ] PR説明に要件の概要が記載されていること
