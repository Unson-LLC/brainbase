# Phase 2: Draft Creation（本文ドラフト作成）

## Mission

**note記事の「本文執筆」を思考フレームワークで自動化**

Phase 1の構成設計に基づき、読者に響く本文を2,500-3,000文字で執筆します。

note-article-writingの7原則とsns-copy-patternsの構文パターンを適用し、**独自性・具体性・プロセスエコノミー**を兼ね備えた記事を生成します。

## Context

**Phase 1の構成設計結果**: `_codex/sns/drafts/{topic}_structure.md`

**参照ファイル**:
- `_codex/sns/note_strategy.md` - note戦略
- `_codex/sns/x_account_profile.md` - アカウント情報・人格

## Task

Phase 1の構成設計に基づき、以下の原則で本文を執筆してください：

### 読まれる記事の7原則

#### 1. 独自性（Original Perspective）

**原則**: 調べたことをそのまま書かない。自分の経験・視点を入れる。

```markdown
# 悪い例
ChatGPTは2022年にリリースされ、多くの企業で活用されています。

# 良い例
ChatGPTを3年使い込んで、僕が辿り着いた結論は「使わない」だった。
```

**適用方法**:
- 「僕は」「私は」で始まる文を意識的に入れる
- 「〜と言われている」を「〜だと僕は思う」に変換
- 一般論を述べた後、「でも僕の場合は〜」で展開

#### 2. 具体性（Specificity）

**原則**: 抽象的な説明ではなく、数字・固有名詞・事例を入れる。

```markdown
# 悪い例
たくさんの業務を自動化しました。

# 良い例
20個の業務を自動化し、週10時間の時間が浮きました。
```

**適用方法**:
- 「たくさん」→ 具体的な数字
- 「いろいろな」→ 具体的な例を3つ列挙
- 「効果があった」→ 定量的な成果（時間、金額、%）

#### 3. ストーリー（Narrative）

**原則**: 情報の羅列ではなく、時系列・因果関係・感情の流れを作る。

```markdown
# 悪い例
自動化ツールAを使い、自動化ツールBを使い、自動化ツールCを使いました。

# 良い例
最初はExcelで管理していた。でも限界が来た。
そこでAを導入した。うまくいった。
調子に乗ってBも導入した。失敗した。
Cに切り替えた。これが正解だった。
```

**適用方法**:
- 「Before → 転機 → After」の流れを作る
- 失敗・試行錯誤を隠さない
- 感情の動き（不安→決断→結果）を描写

#### 4. 継続性（Serialization）

**原則**: 単発記事ではなく、シリーズ化・チャレンジ系にする。

```markdown
# 単発（読者が離脱しやすい）
「AIツール10選」

# シリーズ（読者が継続フォロー）
「100日チャレンジ Day 1」→ Day 2 → Day 3...
```

**適用方法**:
- タイトルに「Day X」「#X」を入れる
- 「次回予告」セクションで次の記事を予告
- 読者が「続きが気になる」終わり方にする

#### 5. タイトル（Headline）

**原則**: 曖昧・長すぎはNG。30-40文字、数字、問いかけ、具体的に。

```markdown
# 悪い例（曖昧）
AIについて思うこと

# 良い例（具体的）
マネジメントから逃げるために、100日間毎日自動化を続けることにした
```

**適用方法**:
- Phase 1で決定したタイトルを使用
- 必要に応じて微調整（本文との整合性）

#### 6. 見出し画像（Header Image）

**原則**: 画像なしはNG。記事内容を視覚化した画像を準備。

**統計**: 見出し画像がある記事はビュー数・スキ数が数十%向上

**適用方法**:
- Phase 3で画像プロンプトを生成
- この段階では本文に集中

#### 7. プロセスエコノミー（Process Economy）

**原則**: 完璧な結果だけ見せず、試行錯誤・失敗・学びを見せる。

```markdown
# 結果だけ（共感されにくい）
100個自動化に成功しました！

# プロセス（共感される）
Day 15: 失敗した。原因は〇〇だった。明日リトライする。
```

**適用方法**:
- 「失敗も公開する理由」セクションを入れる
- 「見えている壁」を先出しする（誠実さの証明）
- 「完璧な成功談」を避ける

### セクション別の書き方

#### 冒頭（これは何か）

**目的**: 読者に「この記事を読む価値」を即座に伝える

**テンプレート**:
```markdown
これは、「〇〇〇」という[チャレンジ/実験/取り組み]の記録です。

なぜこんなことを始めたのか。
結論から言うと、**[逆説的な理由]**からです。
```

**ポイント**:
- 1-2文で「これは何か」を明示
- 「なぜ」を先出しして読者の興味を引く
- 逆説的・意外性のある理由で引き込む

#### 背景・文脈（なぜやるのか）

**目的**: 読者が共感できる文脈を提供

**テンプレート**:
```markdown
## なぜ「[逆説的な表現]」なのか

「[一般的な解釈]」と聞くと、[ネガティブな印象]かもしれません。

でも僕が言いたいのは、**[本当の意図]**ということです。

[職種]として[経験年数]働いてきて、気づいたことがあります。

**[核心となる気づき]**ということ。
```

**ポイント**:
- 逆説的な表現の真意を説明
- 個人の経験から導き出された気づきを共有
- 「〜ということ」で結論を明確化

#### 実績・具体例（証拠）

**目的**: 信頼性を担保する具体的な数字・事例

**テンプレート**:
```markdown
## すでに完了していること

実は、[現在の状況/実績]があります。

例えば：

1. **[具体例1]** - [効果・結果]
2. **[具体例2]** - [効果・結果]
3. **[具体例3]** - [効果・結果]

これだけで、[定量的な成果]が得られました。
```

**ポイント**:
- 具体的な数字を入れる（時間、件数、%等）
- 箇条書きで視認性を高める
- 定量的な成果で説得力を持たせる

#### まとめ（箇条書き）

**目的**: 記事の要点を整理（スキミング読者への配慮）

**テンプレート**:
```markdown
## まとめ

- **チャレンジ内容**: [内容]
- **期間**: [開始日]〜[終了日]
- **すでに完了**: [N]個
- **残り**: [M]個
- **公開ルール**: [ルール]

進捗は[プラットフォーム]で[頻度]報告しています。
```

**ポイント**:
- 箇条書きで要点を5つ程度
- 他プラットフォームへの誘導（X等）
- 継続フォローへの導線

#### 次回予告（継続読者を作る）

**目的**: 「次も読みたい」と思わせる

**テンプレート**:
```markdown
## 次回予告

[時期]からさっそく、[次のアクション]に取り組みます。

最初のターゲットは「[具体的な内容]」。

[Before状態]を、[After状態]に変えます。
うまくいくか、失敗するか——結果は[プラットフォーム]で。
```

**ポイント**:
- 次にやることを具体的に予告
- Before/Afterで期待値を作る
- 結果発表の場所を明示（他プラットフォームへの誘導）

## Important Notes

1. **文字数の重要性**: 2,500-3,000文字が目標。短すぎるとSEO/AIOに弱く、長すぎると読まれない
2. **7原則の適用**: 6つ以上（見出し画像を除く）を意識的に適用
3. **プロセスエコノミー**: 失敗・課題も公開する姿勢を明示
4. **クロスプラットフォーム**: X等への誘導リンクを含める
5. **構成設計の遵守**: Phase 1で決定した構成を尊重

## Success Criteria

- ✅ 2,500文字以上
- ✅ 見出しで構造化されている（#, ##を使用）
- ✅ まとめセクションがある
- ✅ 次回予告がある
- ✅ 7原則のうち5つ以上が適用されている
- ✅ 具体的な数字が3つ以上含まれている
- ✅ 失敗・課題が1つ以上公開されている

このSubagentのドラフトが、Phase 2.5の鬼編集長レビューの対象となります。
