# Phase 2.5: Editor Review（鬼編集長レビュー）

## Mission

**note記事の「品質レビュー」を鬼編集長チェックリストで自動化**

Phase 2で作成されたドラフトを、note-article-writingの鬼編集長チェックリストで厳密に評価します。

80点以上で合格、80点未満はリプラン対象です。

## Context

**Phase 2のドラフト**: `_codex/sns/drafts/{topic}_draft.md`

**参照ファイル**:
- `_codex/sns/note_strategy.md` - note戦略
- `_codex/sns/rules.md` - ガードレール

## Task

以下の4カテゴリ・16項目でドラフトを評価してください：

### 鬼編集長チェックリスト

#### 1. 構成チェック（各5点 × 6項目 = 30点）

| # | 項目 | チェック | 配点 |
|---|------|---------|------|
| 1-1 | 冒頭で「これは何か」を明示している | ✅/❌ | 5点 |
| 1-2 | 2,500文字以上ある | ✅/❌ | 5点 |
| 1-3 | 見出しで構造化されている（##使用） | ✅/❌ | 5点 |
| 1-4 | まとめセクションがある | ✅/❌ | 5点 |
| 1-5 | 次回予告がある | ✅/❌ | 5点 |
| 1-6 | 時系列・年号が正確である | ✅/❌ | 5点 |

**評価基準**:
- 1-1: 最初の200文字以内に「これは〜の記録です」等の説明があるか
- 1-2: 文字数カウント（空白・改行除く）
- 1-3: ##見出しが3つ以上使われているか
- 1-4: 「まとめ」「おわりに」等のセクションがあるか
- 1-5: 「次回」「次は」等の予告があるか
- 1-6: 「20XX年以降」等の時系列表現が執筆時点と矛盾しないか。引用元記事の発表日との整合性があるか。古い情報を現在形で書いていないか

---

#### 2. 独自性チェック（各5点 × 4項目 = 20点）

| # | 項目 | チェック | 配点 |
|---|------|---------|------|
| 2-1 | 自分の経験・視点が入っている | ✅/❌ | 5点 |
| 2-2 | 具体的な数字がある（3つ以上） | ✅/❌ | 5点 |
| 2-3 | 固有名詞がある（ツール名、サービス名等） | ✅/❌ | 5点 |
| 2-4 | 「誰でも書ける内容」になっていない | ✅/❌ | 5点 |

**評価基準**:
- 2-1: 「僕は」「私は」「経験上」等の一人称表現があるか
- 2-2: 数字が3つ以上含まれているか（例: 100日、20個、10時間）
- 2-3: ツール名、サービス名、会社名等の固有名詞があるか
- 2-4: 一般論だけでなく、著者独自の主張・結論があるか

**2-4の詳細評価**:
```
❌ 誰でも書ける例:
「AIを活用すると業務効率化できます」

✅ 独自性がある例:
「PMとして10年働いて気づいた。PMの仕事の8割は人間がやらなくていい」
```

---

#### 3. プロセスエコノミーチェック（各10点 × 3項目 = 30点）

| # | 項目 | チェック | 配点 |
|---|------|---------|------|
| 3-1 | 失敗・課題も公開している | ✅/❌ | 10点 |
| 3-2 | 「完璧な成功談」だけになっていない | ✅/❌ | 10点 |
| 3-3 | 読者が「一緒に成長を見守れる」感覚がある | ✅/❌ | 10点 |

**評価基準**:
- 3-1: 「失敗」「課題」「壁」「難しい」等のネガティブ表現が含まれているか
- 3-2: 全体の論調が「うまくいった！」だけでなく、試行錯誤が見えるか
- 3-3: 「次回」「続き」「一緒に」等、読者を巻き込む表現があるか

**3-2の詳細評価**:
```
❌ 完璧な成功談:
「100個全部自動化に成功しました！」

✅ プロセスが見える:
「実は、顧客対応カテゴリはまだ未着手です。FAQベースの整備が先に必要だと気づいた」
```

---

#### 4. クロスプラットフォームチェック（各5点 × 3項目 = 15点）

| # | 項目 | チェック | 配点 |
|---|------|---------|------|
| 4-1 | X等への誘導リンクがある | ✅/❌ | 5点 |
| 4-2 | note単独でも完結する内容 | ✅/❌ | 5点 |
| 4-3 | 継続フォローへの導線がある | ✅/❌ | 5点 |

**評価基準**:
- 4-1: X（Twitter）のURL、@メンション、またはリンクがあるか
- 4-2: Xを見なくてもnote記事だけで理解できる内容か
- 4-3: 「フォロー」「次回予告」等、継続的な関係構築への導線があるか

---

### スコア基準（ボーナスポイント ≤5点）

**基本点**: 95点満点

**ボーナスポイント**（最大+5点）:
- 逆説的なタイトルがある: +2点
- 感情の動きが描写されている: +2点
- 読者への問いかけがある: +1点

**合格基準**:
- **80点以上**: ✅ 合格（Phase 3へ進行）
- **70-79点**: ⚠️ 要修正（軽微な修正で合格可能）
- **70点未満**: ❌ 不合格（リプラン必須）

## Run Ledger（実行台帳）記録

レビュー結果は **Run Ledger** に必ず記録する。

### 手順
1. 対象ドラフトのfrontmatterから `id` を取得（なければファイル名）
2. スコアと合否を反映してRun Ledgerを更新

### 例（Bash）
```
python _codex/common/ops/scripts/run_ledger.py update \
  --run-id "<id>" \
  --update-json '{"phase":"review","status":"review","decision":"hold","reason":"score<80"}'
```

合格時は:
```
python _codex/common/ops/scripts/run_ledger.py update \
  --run-id "<id>" \
  --update-json '{"phase":"review","status":"approved","decision":"approve"}'
```

## Important Notes

1. **80点以上が合格ライン**: 厳しめの基準だが、品質を担保するため
2. **プロセスエコノミーの重要性**: 30点分を占める最重要カテゴリ
3. **修正提案の具体性**: 抽象的な指摘ではなく、例文まで提示
4. **リプランのトリガー**: 70点未満は自動的にリプラン対象

## Success Criteria

- ✅ スコア80点以上
- ✅ 構成チェック全項目クリア（25/25）
- ✅ 独自性チェック3項目以上クリア（15/20以上）
- ✅ プロセスエコノミーチェック2項目以上クリア（20/30以上）
- ✅ 検出されたNG項目に対する修正提案が具体的

このSubagentのレビュー結果が、Phase 3の最終調整の基盤となります。
