# Phase 1: Target Analysis（見る人の視点）

## Mission

**「見る人の視点」を思考フレームワークで自動分析**

従来の/snsでは「ターゲットが違う」「相手の立場に立って考えて」等の手動修正が頻発していました。

このSubagentは、customer-centric-marketing-n1とx-infra-strategy-dilの思考フレームワークを使い、**「誰が」「なぜ響くか」「どの心理フロー」**を自動分析します。

## Context

**トピック**: [投稿したい内容]

**参照ファイル**:
- `_codex/sns/x_account_profile.md` - アカウント情報・ターゲット層
- `_codex/sns/sns_strategy_os.md` - プロセスエコノミー、ワンチャン感、逆ペルソナ設計

## Task

customer-centric-marketing-n1とx-infra-strategy-dilの思考フレームワークを使い、以下を分析してください：

### 0. トピック抽象化（技術ネタ → 経営課題への変換）

**目的**: GitHub履歴由来の技術的トピックを、経営者が共感できるレベルまで抽象化する。

**判定**: トピックに以下が含まれるか確認:
- git, worktree, API, CI/CD, Lambda, MCP等の技術用語
- 特定ツール名（freee, Airtable, Slack, Notion等）
- コマンドや設定ファイルへの言及

**抽象化が不要なケース**:
- 既に感情・状況ベースで表現されている
- 「経営」「判断」「失敗」「学び」等の普遍的キーワードで構成

**変換ルール**:

| 技術的な表現 | 経営者向け抽象化 | 普遍的な課題 |
|-------------|-----------------|-------------|
| 「worktreeでデータが消えた」 | 「3時間の作業が一瞬で消えた」 | リソース喪失リスク |
| 「freee APIで自動化できなかった」 | 「自動化できると思ってた。大間違いだった」 | 期待と現実のギャップ |
| 「git symlinkの罠」 | 「便利なはずのツールに裏切られた」 | ツール依存のリスク |
| 「CI/CDでエラーなし失敗」 | 「成功したと思ったら、何も動いてなかった」 | 見えないエラーの恐怖 |
| 「1ファイル500行超え」 | 「気づいたら手に負えない状態になってた」 | 複雑性の蓄積 |
| 「788行のサンプルデータ」 | 「誰も使わないかもしれないものに3日かけた」 | 投資対効果の不確実性 |

**抽象化の3ステップ**:
1. **技術用語を削除**: ツール名、コマンド名、専門用語を一般語に置換
2. **感情を付与**: 「詰んだ」「裏切られた」「消えた」など人間の反応を追加
3. **普遍化**: 「誰でも経験しうる状況」に変換

**出力**:
- 元トピック（技術的）
- 抽象化後トピック（経営者向け）
- 以降のStepでは**抽象化後トピック**を使用

### 1. 軸足確認（最優先）

**トピックが軸足の範囲内か確認する**

x_account_profile.md の「専門領域」を確認し、トピックが専門領域に該当するか判定：

**判定基準**:
| トピック例 | 軸足判定 | アクション |
|---------|---------|-----------|
| mana-weekly半自動化 | ✅ 範囲内（実装経験あり） | エッジを効かせてOK |
| AI研究の最先端 | ❌ 範囲外（研究者ではない） | トピック変更 or トーン下げ |
| タスク管理ツール選定 | ✅ 範囲内（brainbase運用実績） | エッジを効かせてOK |
| 医療業界のDX | ❌ 範囲外（医療業界経験なし） | トピック変更 or トーン下げ |

**軸足から外れている場合の対応**:
1. トピックを修正（軸足の範囲内に変更）
2. または発言トーンを下げる（批判→質問、断定→経験談）

**重要**: 軸足から外れた状態でのエッジの効いた発言は、炎上リスク + 信頼失墜を招く

### 2. N=1ペルソナの設定

**customer-centric-marketing-n1の思考法を適用**:
- このトピックに**最も響く**たった一人の顧客を想定
- 実在する顧客層から具体的な1人を設定
- ロイヤル顧客？一般顧客？認知・未購買層？離反顧客？

**具体化すべき項目**:
- 役職・立場（例: 経営者、事業責任者、GM候補）
- 現在の課題（例: 技術チームがいない、運用不安、セキュリティ不安）
- このアカウントへの認知度（既知？未知？）
- ブランド選好（積極的？消極的？中立？）

### 3. 9セグマップ上の位置を特定

**5セグ（顧客ピラミッド）**:
1. ロイヤル顧客
2. 一般顧客
3. 離反顧客
4. 認知・未購買顧客
5. 未認知顧客

**9セグ（+ ブランド選好）**:
- 積極的（次回も購買意向が高い）
- 中立（どちらでもない）
- 消極的（次回購買意向が低い）

**判断基準**:
- このトピックは既存顧客向け？新規顧客向け？
- ブランド選好を上げる内容？下げるリスクがある内容？

### 4. 顧客心理フローの判定

**x-infra-strategy-dilの思考法を適用**:

Xを集客装置ではなく「接点設計」として捉え、**認知→理解→信頼→行動**のどの段階を狙うか判断：

| 段階 | 目的 | 投稿の役割 |
|------|------|-----------|
| 認知 | 第一想起の獲得 | 「見慣れ」を作る、記憶に残る |
| 理解 | 価値理解・差別化 | 何をする人か、なぜ信頼できるか |
| 信頼 | 不安解消 | 「この人に任せて大丈夫か」を証明 |
| 行動 | 商談・採用・DM | 具体的なアクション喚起 |

**このトピックは顧客心理フローのどこを狙うべきか？**

### 5. ペルソナの心理状態を推測

**「悩み」「判断（きっかけ）」「結果」**を推測：

- **悩み**: このペルソナが抱えている課題・不安は？
  - 例: 「自分で作ったプロトタイプ、セキュリティが心配」
  - 例: 「技術チームがいない状態でリリースして大丈夫？」

- **判断（きっかけ）**: どんなきっかけで行動を起こすか？
  - 例: 「継続メンテナンスが必要だと気づいた」
  - 例: 「商用レベルに引き上げる必要性を感じた」

- **結果**: 行動した結果どうなるか？何を期待しているか？
  - 例: 「技術チームを外部に持てる安心感」
  - 例: 「セキュリティ診断・コードレビューで品質保証」

### 6. プロセスエコノミー観点の評価

**sns_strategy_os.mdの思考法を適用**:

このトピックは「過程」を見せる価値があるか？

- ✅ 良い例: 失敗・試行錯誤があり、「次どうなる？」がある
- ❌ 悪い例: 1話完結、結果だけ、情報提供のみ

**評価項目**:
- プロセスエコノミー: 「過程」が見えるか？
- ワンチャン感: 挑戦的だが不可能ではない目標か？
- 誤解→共感→回収: 「事件化」できるか？

### 7. 推奨構文の選定

sns-copy-patternsから、このペルソナに響く構文を推奨：

- **階段構文（ロードマップ型）**: 学習順序、ツール導入順序を示す
- **さとり構文（両面訴求型）**: ターゲット層と非ターゲット層の両方に訴求
- **誤解→共感→回収（ストーリー型）**: 冒頭で誤解を生み、共感でつなぎ、回収で納得させる

**判断基準**:
- ストーリー性がある → 誤解→共感→回収
- ノウハウ・手順を伝える → 階段構文
- 幅広い層に訴求したい → さとり構文

## Important Notes

1. **N=1の重要性**: 統計的な平均値ではなく、**たった一人の深層心理**を掘り下げる
2. **思考フレームワークの活用**: customer-centric-marketing-n1とx-infra-strategy-dilを「How to think」として使う
3. **具体性**: 「経営者」ではなく「自分でプロトタイプを作った経営者、技術チームがいない、運用不安あり」のように具体化
4. **逆ペルソナチェック**: 「AIに興味がない人」でも離脱しない訴求か？

## Success Criteria

- ✅ トピックが技術的な場合、経営者向けに抽象化されている
- ✅ N=1ペルソナが具体的に設定されている（役職・立場・課題が明確）
- ✅ 9セグマップ上の位置が判断されている
- ✅ 顧客心理フロー（認知→理解→信頼→行動）が判定されている
- ✅ 「悩み」「判断」「結果」が推測されている
- ✅ 訴求ポイントが明確化されている
- ✅ 推奨構文が選定されている（理由付き）

このSubagentの分析結果が、Phase 2のドラフト作成の基盤となります。
