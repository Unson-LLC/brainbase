## 1. PMの実践とは

### PMの3つの責任

1. **チームメンバーのベストを引き出す**
2. **直属チーム以外の人と一緒に働く**
3. **曖昧さに対処する**

> 「何かがわからない状況を把握するスキルは、その後で何かをするスキルと同じくらい重要」

### PMではないこと

| 誤解 | 現実 |
|------|------|
| ボス・ミニCEO | 権限なき責任を持つ存在 |
| 天才発明家 | 人やものをつなげる調整役 |
| 指示待ち | 自ら課題を見つけて動く |

### 悪いPMのパターン

1. **ジャーゴンジョッキー** - 専門用語で煙に巻く
2. **スティーブ・ジョブズ信奉者** - ビジョナリーを気取りユーザーの声を無視
3. **英雄のPM** - 自分だけで全部やろうとする
4. **張り屋さん** - 達成より働いた時間を誇示
5. **殉教者** - 責任転嫁と言い訳に終始

## 2. COREスキル

PMに必要な4つの基本スキル：

```
┌─────────────────────────────────────┐
│           CORE Skills               │
├──────────┬──────────┬───────────────┤
│Communication│Organization│Research │
│ コミュニケーション│ 組織化 │ リサーチ   │
├──────────┴──────────┴───────────────┤
│              Execution               │
│                実行                  │
└─────────────────────────────────────┘
```

### 2.1 コミュニケーション

**核心**: 明快さと心地よさのバランス

- **明快さ**: 曖昧な表現を避け、具体的に伝える
- **心地よさ**: 相手の立場を尊重し、対立を恐れない

**自己チェック**:
- やりにくい会話をファシリテーションしているか？
- 遠回しではなく単刀直入に伝えているか？
- 相手の発言に熟慮深く反応しているか？

### 2.2 組織化

**核心**: 自らを不要とせよ

- チームが自律的に動ける仕組みを作る
- ボトルネックにならない
- 情報とプロセスを整備する

**自己チェック**:
- 1週間休んでも優先順位付けとデリバリーができるか？
- 「何をしているのか？なぜ？」に誰でも答えられるか？
- チームに必要な情報に簡単にたどり着けるか？

### 2.3 リサーチ

**核心**: 好奇心をもって生きよ

- ユーザーとの対話を継続する
- 市場の変化を常に把握する
- 仮説を修正する新しい情報を探す

**自己チェック**:
- 週1回以上ユーザーと話しているか？
- 意思決定はビジネスゴールとユーザーニーズに基づいているか？
- ユーザーニーズは実際のユーザーの声を反映しているか？

### 2.4 実行

**核心**: すべての努力はアウトカムのために

- アウトプット（作ること）よりアウトカム（成果）を重視
- 戦略的ゴールと日々の努力をつなげる
- 「忙しいこと」で正当化しない

**自己チェック**:
- ビジネスインパクトへの複数の道筋を持っているか？
- 戦略的ゴールがスプリント計画の中心になっているか？
- 作業量の積み上げで正当化していないか？

## 3. 好奇心をあらわにする

### しなやかマインドセット

- 固定マインドセット: 「自分は〇〇な人間だ」
- しなやかマインドセット: 「今はできないが学べる」

### 実践のポイント

1. **違いという贈り物** - 異なる意見を学びの機会と捉える
2. **距離を置く** - 自分のアイデアと自己を切り離す
3. **「なぜ」を訊ねる** - 理由を訊くことで理解を深める
4. **決めつけない** - 答えを持たない勇気

## 4. コミュニケーション技術

### 原則

1. **あたりまえを問う** - 暗黙の前提を明確にする
2. **遠回しではなく単刀直入に** - 曖昧さを排除する
3. **アウトカムは意図より重要** - 結果で判断される
4. **「よさそう」に注意** - 安易な同意は問題の先送り

### Disagree & Commit 戦術

```
┌─────────────────────────────────────┐
│  1. 異議を表明する（明確に）         │
│  2. 理由を説明する                   │
│  3. 決定が下されたらコミットする      │
│  4. 決定後は全力で推進する           │
└─────────────────────────────────────┘
```

### やってはいけないこと

- 陰で不満を言う
- 決定後に足を引っ張る
- 「言ったとおりになった」と後出し

## 5. シニアステークホルダーと働く

### 「権力」から情報へ

- 上位者の発言を鵜呑みにしない
- 「なぜ」を理解する
- 情報として受け止め、分析する

### 気に入らない答えでも答えは答え

- 明確な「No」は前に進める
- 曖昧な返答より良い
- 次のアクションを決められる

### 「いいね、やって」の罠

**問題**: 上位者が安易にGOを出すと、チームが混乱する

**対処法**:
1. 本当に優先すべきか確認
2. 既存の計画との整合性を問う
3. 何を止めるべきか明確にする

### 上にマネジメントする

- 上司の成功を助ける
- 上司の上司の視点を理解する
- 相手の関心事に寄り添う

## 6. ユーザーに話しかける

### ステークホルダー ≠ ユーザー

| ステークホルダー | ユーザー |
|------------------|----------|
| 社内の関係者 | 実際にプロダクトを使う人 |
| ビジネス目標を持つ | 自分の課題を解決したい |
| 発言力がある | 声が届きにくいことも |

### ペルソナの罠

- 架空のペルソナより実在のユーザー
- データよりも対話
- 「典型的なユーザー」は存在しない

### リサーチのベストプラクティス

1. **週1回以上**ユーザーと話す
2. **行動を観察**する（言葉だけでなく）
3. **オープンな質問**をする
4. **仮説を持ちつつ**修正する準備をする

---

## 7. 「ベストプラクティス」の限界

### 主張を鵜呑みにしない

- 「他社もやっている」は根拠にならない
- コンテキストが違えば結果も違う
- 成功事例の裏に失敗がある

### フレームワークは有用なフィクション

- 完璧なフレームワークは存在しない
- 出発点として使う
- 自社に合わせてカスタマイズする

### 「わからない」と働く

- 正解がないことを受け入れる
- 実験と学習のサイクルを回す
- 完璧を求めず前に進む

## 8. アジャイルの実践

### アジャイルにまつわる3つの迷信

1. **アジャイル＝スクラム** → 方法論は手段
2. **アジャイル＝計画なし** → 適応的計画は計画
3. **アジャイル＝速い** → 学習を重視

### マニフェストからモンスターへ

- 形式的なアジャイルは本末転倒
- プロセスより人を重視
- 「アジャイルをやる」より「アジャイルである」

### 組織への適合

```
┌─────────────────────────────────────┐
│ スタートアップ：軽量・柔軟          │
│ 成長期：プロセスの標準化            │
│ 大企業：ガバナンスとのバランス       │
└─────────────────────────────────────┘
```

## 9. ドキュメントとロードマップ

### 「俺たちは要件の奴隷だ！」

- 仕様書はプロダクトではない
- ドキュメントは手段
- 過剰なドキュメントは害になる

### ガントチャートの幻想

- 計画通りに進むことは稀
- 詳細な計画より方向性
- 柔軟性を持たせる

### 安全なドキュメント

- **Living Document**: 常に更新される
- **SSOT（Single Source of Truth）**: 唯一の真実の情報源
- **最初のドラフトは1ページ**: 詳細は後から

### ロードマップの原則

1. **方向性を示す**（詳細な計画ではない）
2. **コミットメントではない**（可能性を示す）
3. **定期的に見直す**

## 10. ビジョン・ミッション・戦略

### アウトカムとアウトプットのシーソー

| アウトプット | アウトカム |
|--------------|------------|
| 機能、リリース | ユーザー価値、ビジネス成果 |
| 測定しやすい | 測定しにくい |
| 短期的 | 長期的 |

### 戦略はシンプルで明快に

**良い戦略の特徴**:
- 1文で説明できる
- 優先順位が明確
- 何をしないかが分かる

### OKR/KPIの活用

- **Objective**: 定性的な目標
- **Key Results**: 定量的な指標
- 達成度より学習を重視

## 11. データ駆動の罠

### 指標をめぐるトラブル

- **虚栄の指標（Vanity Metrics）**: 見栄えは良いが意思決定に使えない
- **グッドハートの法則**: 指標が目標になると指標でなくなる

### 指標の使い方

1. **行動につながる指標**を選ぶ
2. **複数の指標**でバランスを取る
3. **定性データ**と組み合わせる

### 王道なんてない

- データは答えを出さない
- 解釈が必要
- 判断はPMの仕事

## 12. 優先順位付け

### すべてを食べることはできない

- 「全部やる」は何もやらないと同じ
- トレードオフを明確に
- 「No」を言う勇気

### 決定もトレードオフ

- 完璧な決定は存在しない
- 決めないことのコスト
- 間違えたら修正する

### フレームワーク

```
┌─────────────────────────────────────┐
│ RICE: Reach × Impact × Confidence  │
│       ────────────────────────     │
│              Effort                 │
├─────────────────────────────────────┤
│ ICE: Impact × Confidence × Ease    │
├─────────────────────────────────────┤
│ MoSCoW: Must/Should/Could/Won't    │
└─────────────────────────────────────┘
```

### 考え方

- **夢は大きく、スタートは小さく**
- **可逆的な決定**は素早く
- **不可逆的な決定**は慎重に

## 13. 分散チームの困難

### 遠くから信頼を築く

- **オーバーコミュニケーション**を心がける
- **非同期コミュニケーション**のルールを決める
- **対面の機会**を大切にする

### コミュニケーションアグリーメント

```yaml
# 例
緊急連絡: Slack DM or 電話
通常連絡: Slack チャンネル
ドキュメント共有: Notion
ミーティング: Google Meet
返信期待時間:
  Slack: 4時間以内
  メール: 24時間以内
```

### 期待を設定する

- **明文化**する
- **確認**する
- **調整**する

## 14. プロダクトリーダーシップ

### 出世することの意味

- ICからマネジメントへ
- 自分で作ることから、人を通じて成果を出すことへ
- スキルセットの変化

### 小さなガードレール、小さなフィードバックループ

- マイクロマネジメントを避ける
- 自律性を尊重する
- 早めに軌道修正する

### リーダーの責任

1. **ビジョンを示す**
2. **障害を取り除く**
3. **成長を支援する**

## 15. 良いときと悪いとき

### 自動操縦の心地よい静寂

- うまくいっているときは静か
- 問題がないことに気づきにくい
- 調子の良いときこそ振り返る

### 世界の重荷を背負う

- すべてを背負わない
- 助けを求める
- 自分を大切にする

### 辛いがその価値はある

- PMは孤独な仕事
- でも、成長の機会は多い
- 影響を与えられる喜び
