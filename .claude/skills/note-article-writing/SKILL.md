## 最適な文字数

| 目的 | 推奨文字数 | 理由 |
|------|-----------|------|
| 読みやすさ重視 | 1,000-2,000文字 | 3分以内で読める |
| SEO/AIO重視 | 3,000文字以上 | 検索エンジンが網羅性を評価 |
| 編集部おすすめ狙い | 2,500-3,000文字 | プロが読みやすいと感じる長さ |
| **推奨** | **2,500-3,000文字** | バランス最適 |

## 記事構成（必須セクション）

### 1. 冒頭（これは何か）

**目的**: 読者に「この記事を読む価値」を即座に伝える

```markdown
これは、「〇〇〇」という[チャレンジ/実験/取り組み]の記録です。

なぜこんなことを始めたのか。
結論から言うと、**[逆説的な理由]**からです。
```

**ポイント**:
- 1-2文で「これは何か」を明示
- 「なぜ」を先出しして読者の興味を引く
- 逆説的・意外性のある理由で引き込む

---

### 2. 背景・文脈（なぜやるのか）

**目的**: 読者が共感できる文脈を提供

```markdown
## なぜ「[逆説的な表現]」なのか

「[一般的な解釈]」と聞くと、[ネガティブな印象]かもしれません。

でも僕が言いたいのは、**[本当の意図]**ということです。

[職種]として[経験年数]働いてきて、気づいたことがあります。

**[核心となる気づき]**ということ。
```

**ポイント**:
- 逆説的な表現の真意を説明
- 個人の経験から導き出された気づきを共有
- 「〜ということ」で結論を明確化

---

### 3. 実績・具体例（証拠）

**目的**: 信頼性を担保する具体的な数字・事例

```markdown
## すでに完了していること

実は、[現在の状況/実績]があります。

例えば：

1. **[具体例1]** - [効果・結果]
2. **[具体例2]** - [効果・結果]
3. **[具体例3]** - [効果・結果]
4. **[具体例4]** - [効果・結果]

これだけで、[定量的な成果]が得られました。
```

**ポイント**:
- 具体的な数字を入れる（時間、件数、%等）
- 箇条書きで視認性を高める
- 定量的な成果で説得力を持たせる

---

### 4. 全体像・計画（何をするのか）

**目的**: 読者に「これから何が起こるか」を見せる

```markdown
## [期間]で[目標]、何を[動詞]のか

[対象]を[N]カテゴリに分類しました。

| カテゴリ | 例 | 難易度 |
|---------|-----|--------|
| [カテゴリ1] | [例1] | ★★☆ |
| [カテゴリ2] | [例2] | ★★★ |
| ... | ... | ... |
```

**ポイント**:
- 表形式で全体像を可視化
- 難易度・優先度など付加情報を追加
- 読者が「全体感」を把握できるようにする

---

### 5. 姿勢・ルール（失敗も公開）

**目的**: プロセスエコノミーの姿勢を明示

```markdown
## 失敗も公開する理由

このチャレンジには、一つルールがあります。

**失敗も遅延もすべて公開する。**

「[成功だけ見せる]」という結果だけを見せるのは簡単です。
でも、それでは誰の役にも立ちません。

実際、すでに見えている壁があります。

- **[壁1]**: [理由]
- **[壁2]**: [理由]

なぜなら、**プロセスを見せることに価値がある**と信じているからです。
```

**ポイント**:
- 失敗・遅延を「恥」ではなく「価値」として扱う
- 具体的な壁・課題を先出しする（誠実さの証明）
- プロセスエコノミーの哲学を明示

---

### 6. 技術・手法（透明性）

**目的**: 再現可能性・信頼性を担保

```markdown
## 技術スタック / 使用ツール

興味がある人のために、使っている[ツール/手法]を共有します。

- **[ツール1]**: [用途]
- **[ツール2]**: [用途]
- **[ツール3]**: [用途]

特別なツールは使っていません。
誰でも手に入るもので、地道に[動詞]を積み上げています。
```

**ポイント**:
- 具体的なツール名を明記
- 「誰でもできる」感を演出（敷居を下げる）
- 専門家向けの情報も提供（ターゲットの拡大）

---

### 7. ビジョン・哲学（独自性）

**目的**: 他の記事にはない「自分だけの視点」を提供

```markdown
## 「[キャッチコピー]」とはどういうことか

僕のキャッチコピーは「[キャッチコピー]」です。

これは冗談ではありません。本気でそう思っています。

[領域]の本質は「[本質]」です。
でも、[本質]の前には膨大な「[前工程]」があります。

その[X]%は、[代替手段]にやらせた方がいい。
人間は、最後の[Y]%——[本当に必要な部分]——だけに集中すればいい。

[期間]後、[主語]がどう変わっているか。楽しみにしていてください。
```

**ポイント**:
- キャッチコピー/軸足との接続
- 具体的な数字で主張を補強（80/20等）
- 未来への期待を持たせる締め

---

### 8. まとめ（箇条書き）

**目的**: 記事の要点を整理（スキミング読者への配慮）

```markdown
## まとめ

- **チャレンジ内容**: [内容]
- **期間**: [開始日]〜[終了日]
- **すでに完了**: [N]個
- **残り**: [M]個
- **公開ルール**: [ルール]

進捗は[プラットフォーム]で[頻度]報告しています。
```

**ポイント**:
- 箇条書きで要点を5つ程度
- 他プラットフォームへの誘導（X等）
- 継続フォローへの導線

---

### 9. 次回予告（継続読者を作る）

**目的**: 「次も読みたい」と思わせる

```markdown
## 次回予告

[時期]からさっそく、[次のアクション]に取り組みます。

最初のターゲットは「[具体的な内容]」。

[Before状態]を、[After状態]に変えます。
うまくいくか、失敗するか——結果は[プラットフォーム]で。
```

**ポイント**:
- 次にやることを具体的に予告
- Before/Afterで期待値を作る
- 結果発表の場所を明示（他プラットフォームへの誘導）

## 読まれる記事の7原則

### 1. 独自性（Original Perspective）

❌ 調べたことをそのまま書く
✅ 自分の経験・視点を入れる

```markdown
# 悪い例
ChatGPTは2022年にリリースされ、多くの企業で活用されています。

# 良い例
ChatGPTを3年使い込んで、僕が辿り着いた結論は「使わない」だった。
```

### 2. 具体性（Specificity）

❌ 抽象的な説明
✅ 数字・固有名詞・事例

```markdown
# 悪い例
たくさんの業務を自動化しました。

# 良い例
20個の業務を自動化し、週10時間の時間が浮きました。
```

### 3. ストーリー（Narrative）

❌ 情報の羅列
✅ 時系列・因果関係・感情の流れ

```markdown
# 悪い例
自動化ツールAを使い、自動化ツールBを使い、自動化ツールCを使いました。

# 良い例
最初はExcelで管理していた。でも限界が来た。
そこでAを導入した。うまくいった。
調子に乗ってBも導入した。失敗した。
Cに切り替えた。これが正解だった。
```

### 4. 継続性（Serialization）

❌ 単発記事
✅ シリーズ化・チャレンジ系

```markdown
# 単発（読者が離脱しやすい）
「AIツール10選」

# シリーズ（読者が継続フォロー）
「100日チャレンジ Day 1」→ Day 2 → Day 3...
```

### 5. タイトル（Headline）

❌ 曖昧・長すぎ
✅ 30-40文字、数字、問いかけ、具体的

```markdown
# 悪い例（曖昧）
AIについて思うこと

# 良い例（具体的）
マネジメントから逃げるために、100日間毎日自動化を続けることにした
```

### 6. 見出し画像（Header Image）

❌ 画像なし
✅ 記事内容を視覚化した画像

**統計**: 見出し画像がある記事はビュー数・スキ数が数十%向上

### 7. プロセスエコノミー（Process Economy）

❌ 完璧な結果だけ見せる
✅ 試行錯誤・失敗・学びを見せる

```markdown
# 結果だけ（共感されにくい）
100個自動化に成功しました！

# プロセス（共感される）
Day 15: 失敗した。原因は〇〇だった。明日リトライする。
```

## 鬼編集長チェックリスト

note記事公開前に確認：

### 構成チェック
- [ ] 冒頭で「これは何か」を明示している
- [ ] 2,500文字以上ある
- [ ] 見出しで構造化されている
- [ ] まとめセクションがある
- [ ] 次回予告がある

### 独自性チェック
- [ ] 自分の経験・視点が入っている
- [ ] 具体的な数字がある
- [ ] 固有名詞（ツール名等）がある
- [ ] 「誰でも書ける内容」になっていない

### プロセスエコノミーチェック
- [ ] 失敗・課題も公開している
- [ ] 「完璧な成功談」だけになっていない
- [ ] 読者が「一緒に成長を見守れる」感覚がある

### クロスプラットフォームチェック
- [ ] X等への誘導リンクがある
- [ ] 他プラットフォームとの役割分担が明確
- [ ] note単独でも完結する内容

## AIO（AI検索最適化）

### note記事がAIOに強い理由

1. **ドメイン信頼性**: note.comは高品質コンテンツプラットフォームとして認知
2. **構造化**: 見出し・セクションがAIに解釈されやすい
3. **専門性**: 長文で深掘りすることで専門性が評価される

### AIO最適化のポイント

```markdown
# 良い例（AIO対応）
## PM業務自動化とは

PM業務自動化とは、プロジェクトマネージャーの定型業務をAIやスクリプトで
自動実行する取り組みです。具体的には以下の業務が対象です：

- 日次タスク一覧の自動生成
- 会議議事録の自動生成
- 進捗レポートの自動作成
```

**ポイント**:
- 見出しに検索キーワードを含める
- 「〇〇とは」形式で定義を明示
- 箇条書きで具体例を列挙
