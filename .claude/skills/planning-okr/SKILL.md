## Triggers

以下の状況で使用：
- プロジェクトの計画を立てるとき
- マイルストーンを設定するとき（milestone-management）
- OKRを設計するとき
- 目標設定が必要なとき
- 四半期・年次計画を作成するとき
- プロジェクト立ち上げ時（project-onboarding）

## プランニングの本質

**原則**: 計画のアウトプットは「計画書」ではなく「行動」

**計画の目的**:
- 将来の需要を予測
- 現状とのギャップを明確化
- ギャップを埋める具体的な行動を決める

**重要な点**:
- 計画書を作ること自体が目的ではない
- 実行すべき具体的なタスク一覧が本当のアウトプット
- 柔軟に修正できる計画であること

**思考パターン**:
- この計画から具体的なタスク一覧が作れるか？
- 計画を実行に移せる状態か？
- 環境が変化したときに柔軟に修正できるか？

## プランニング・プロセス（3ステップ）

### ステップ1: 将来の需要を予測

**目的**: 環境の変化を予測し、将来の目標を設定

**予測する項目**:
- 市場の動向
- 技術の進化
- 競合の動き
- 顧客ニーズの変化
- 法規制の変更

**予測の方法**:
- 過去データの分析
- トレンドの特定
- 専門家の意見
- 顧客インタビュー

**例**:
```
## 将来の需要予測（2026年Q1）

### 市場動向
- AIツールの普及加速（前年比150%成長予測）
- リモートワーク継続（オフィス回帰は30%以下）

### 顧客ニーズ
- 自動化への需要増加
- セキュリティ要件の厳格化

### 技術進化
- LLMの精度向上
- マルチモーダルAIの普及
```

### ステップ2: 現状を把握

**目的**: 現在の能力と仕掛かりプロジェクトを把握し、何もしなければ将来どうなるかを予測

**把握する項目**:
- 現在の能力（チーム、リソース、スキル）
- 仕掛かりプロジェクト（進行中の取り組み）
- 何もしなければどうなるか（現状維持の場合）

**例**:
```
## 現状把握（2025年Q4）

### 現在の能力
- チーム: 5名（エンジニア3名、デザイナー1名、PM1名）
- スキル: React, Node.js, AWS
- リソース: 月400時間（80時間/人 × 5名）

### 仕掛かりプロジェクト
- プロジェクトA: 進行中（残り2ヶ月）
- プロジェクトB: 計画中（開始予定: 2026年Q1）

### 何もしなければ
- プロジェクトAのみ完了
- 新規機能開発は停滞
- 技術負債が蓄積
```

### ステップ3: ギャップを埋める行動を決める

**目的**: 需要と現状のギャップを特定し、具体的な行動を決める

**ギャップ分析**:
```
需要（2026年Q1）   現状（2025年Q4）    ギャップ
─────────────────────────────────────────
AIツール提供       基本機能のみ         自動化機能不足
チーム8名         チーム5名            3名不足
```

**行動計画**:
```
## 行動計画（2025年Q4 → 2026年Q1）

### 採用
- [ ] エンジニア2名採用（期限: 2026年1月）
- [ ] デザイナー1名採用（期限: 2026年2月）

### 開発
- [ ] AI自動化機能の開発（期限: 2026年3月）
- [ ] セキュリティ強化（期限: 2026年2月）

### トレーニング
- [ ] チーム全員にAI研修（期限: 2026年1月）
```

**思考パターン**:
- 将来の需要を過去のデータから予測しているか？
- 現状の能力を正確に把握しているか？
- ギャップを埋める具体的な行動を決めているか？

## 目標管理（MBO/OKR）

### Objective（どこへ行きたいか）

**定性的な目標**:
- 方向性を示す
- チームを鼓舞する
- 野心的である

**良いObjectiveの例**:
- ✅ 「顧客満足度を向上させる」
- ✅ 「市場シェアNo.1を目指す」
- ✅ 「最高のプロダクトを作る」

**悪いObjectiveの例**:
- ❌ 「売上を10%増やす」（定量的すぎる、KRに移すべき）
- ❌ 「頑張る」（抽象的すぎる）

### Key Results（ペースをどう決めるか）

**定量的な指標**:
- 進捗を測定
- 達成度を可視化
- 3-5個に絞る

**良いKey Resultsの例**:
- ✅ 「NPS（Net Promoter Score）を+10ポイント向上」
- ✅ 「サポート応答時間を24時間以内に」
- ✅ 「顧客継続率を90%以上に」

**悪いKey Resultsの例**:
- ❌ 「顧客満足度を向上させる」（定量化されていない、Objectiveに移すべき）
- ❌ 「営業訪問件数100件」（活動指標、成果指標ではない）

### OKRの設定

**フォーマット**:
```
Objective: [定性的な目標]
Key Results:
├─ KR1: [定量的指標1]
├─ KR2: [定量的指標2]
└─ KR3: [定量的指標3]
```

**例**:
```
Objective: 顧客満足度を向上させる
Key Results:
├─ NPS を+10ポイント向上（現状30 → 目標40）
├─ サポート応答時間を24時間以内に（現状48時間）
└─ 顧客継続率を90%以上に（現状85%）
```

**思考パターン**:
- Objectiveは定性的で野心的か？
- Key Resultsは定量的で測定可能か？
- KRは3-5個に絞られているか？

## OKRの設計プロセス

### ステップ1: Objectiveを決める

**問い**:
- 「何を達成したいか？」
- 「どこへ向かいたいか？」
- 「チームの最優先事項は何か？」

**ブレインストーミング**:
- チームでObjectiveを考える
- 複数の候補を出す
- 1つに絞る

### ステップ2: Key Resultsを決める

**問い**:
- 「Objectiveを達成したことをどう測るか？」
- 「どの指標が重要か？」
- 「どこまで達成すれば成功か？」

**指標の選択**:
- 成果指標（活動指標ではない）
- 測定可能
- 影響力が大きい

### ステップ3: 現状値と目標値を設定

**現状値**:
- 今どこにいるか？
- データを確認

**目標値**:
- どこまで行きたいか？
- 野心的だが達成可能

**ストレッチゴール**:
- 70-80%の達成率を目指す
- 100%達成可能な目標は低すぎる

### ステップ4: 行動計画に落とす

**問い**:
- 「KRを達成するために何をするか？」
- 「具体的なタスクは何か？」

**タスク化**:
```
KR1: NPS を+10ポイント向上
└─ タスク
    ├─ [ ] 顧客アンケート実施（毎月）
    ├─ [ ] フィードバックの分析と改善（毎週）
    └─ [ ] サポート品質向上施策（継続）
```

## 計画期間と再計画

**計画期間**:
- 四半期（3ヶ月）: OKR
- 年次（12ヶ月）: 年間目標
- 3年: 長期ビジョン

**再計画**:
- 環境が変化したら柔軟に修正
- 四半期ごとにOKRを見直し
- 固執しない

**思考パターン**:
- 計画期間は適切か？
- 環境が変化したときに修正しているか？
- 計画に固執していないか？

## OKRの運用

### 四半期サイクル

**Week 0: OKR設定**
- チームでObjectiveとKRを決める
- 現状値と目標値を設定
- 行動計画に落とす

**Week 1-11: 実行とモニタリング**
- 毎週KRの進捗を確認
- 障害があればエスカレーション
- 必要に応じて行動計画を修正

**Week 12: レビューと次四半期の準備**
- OKRの達成度を確認
- 振り返り（何が効いたか、効かなかったか）
- 次四半期のOKRを設定

### 週次チェック

**問い**:
- KRの進捗はどうか？
- 目標達成のペースは適切か？
- 障害は何か？

**フォーマット例**:
```
## OKR週次チェック（Week 5）

### KR1: NPS を+10ポイント向上（目標: 30 → 40）
- 現状値: 33（+3ポイント）
- ペース: やや遅れ（Week 5時点で+5ポイント必要）
- 障害: フィードバックの分析が遅れている
- アクション: 分析リソースを追加

### KR2: サポート応答時間を24時間以内に（目標: 48時間 → 24時間）
- 現状値: 36時間
- ペース: 順調
- 障害: なし
```

## OKRの失敗パターン

### 失敗1: 活動指標をKRにする

**症状**:
- 「営業訪問件数100件」
- 「会議10回開催」

**対処**:
- 成果指標に変える
- 「取ってきた注文件数20件」
- 「意思決定5件完了」

### 失敗2: KRが多すぎる

**症状**:
- KRが10個以上
- フォーカスが分散
- 達成できない

**対処**:
- 3-5個に絞る
- 優先順位をつける
- 最重要なKRに集中

### 失敗3: 100%達成可能な目標

**症状**:
- 簡単すぎる目標
- チームが挑戦しない
- 成長しない

**対処**:
- ストレッチゴールを設定
- 70-80%の達成率を目指す
- 野心的な目標

### 失敗4: OKRを設定して放置

**症状**:
- 週次チェックがない
- 進捗を測定しない
- 四半期末に確認するだけ

**対処**:
- 毎週KRの進捗を確認
- ダッシュボード化
- 1on1でフォロー
