## Triggers

以下の状況で使用：
- SaaS/AIプロダクトのロードマップを策定したいとき
- MVP→PMFまでの段階的フィット検証を設計したいとき
- OKR/KPIでチーム共通のマイルストーンを設定したいとき
- 「今はやらない」機能の切り分け基準を決めたいとき
- クロスチーム（エンジニア・CS・BizDev）の検証サイクルを設計したいとき

## 1. プロダクト開発の優先順位付け（MVP〜PMFまで）

### 基本原則
- 限られたリソースを最大限に活用するため、**プロダクト開発の優先順位付け**が極めて重要
- まずは顧客にとって本質的価値のある**MVP（Minimum Viable Product）**の構築に集中し、余計な機能は削ぎ落とす
- MVPではターゲットユーザーの「なくてはならない」機能だけにフォーカスし、できるだけ早く市場からフィードバックを得る
- **インパクトの大きい機能を優先実装**することが成功への近道

### 段階的フィット（Fit）フレームワーク

**スタートアップ・フィット・ジャーニー**として以下の順序で検証を進める：

| フェーズ | 名称 | 検証内容 | ゴール |
|---------|------|---------|--------|
| 1 | **CPF** (Customer/Problem Fit) | 想定顧客の抱える課題が本当に存在するか | 課題が十分に重要で解決が求められているか確認 |
| 2 | **PSF** (Problem/Solution Fit) | 課題に対する解決策の仮説が顧客に受け入れられるか | ユーザーインタビューやPoCでソリューションの妥当性をテスト |
| 3 | **SPF** (Solution/Product Fit) | ソリューションを実際のプロダクトとして磨き上げられるか | α版・β版提供によりユーザー定着度を観察 |
| 4 | **PMF** (Product/Market Fit) | プロダクトが市場にフィットし「勝手に広がっていく」状態か | ユーザーが価値に熱狂し、口コミや継続利用で自然と成長軌道に乗る |

### フレームワークの効果
- 「今は何をゴールにすべきか」「次に控える課題は何か」を全員が明確に理解できる
- 「いきなり売上拡大を狙って営業しまくる」といった早計な行動を避けられる
- **不確実性の高い順に検証を進める文化**が醸成される

## 2. チーム共通のマイルストーン設計（OKRや指標の活用）

### OKRによる目標設定
- プロダクトロードマップ上の施策項目を「**Objective**（何を達成するか）」として定義
- 成功基準を「**Key Result**」として数値目標で設定

**例：**
- Objective: β版ユーザーの継続率向上
- Key Result: 継続率を現在の20%から50%に改善する

### ロードマップ設計のベストプラクティス

| 項目 | 推奨 |
|------|------|
| 計画期間 | **3〜6ヶ月**の短いスパン（1年先の詳細は決め打ちしない） |
| 更新頻度 | 学びに応じて柔軟に更新 |
| 共通認識 | 「ロードマップは**仮説であり変化しうる**」 |
| 指標設計 | **数値指標（KPI）とフィット段階を組み合わせ**て定義 |

### 指標設計の例
- 「○○機能のユーザー有効活用率70%達成（**PSF段階クリア**）」
- 「解約率月5%以下で安定（**PMF段階近づく**）」

## 3. クロスチームでのフェーズ設計と検証サイクル

### Build-Measure-Learn サイクル
- **実行（Build）と検証（Measure/Learn）をセット**で回す文化を醸成
- ビルド→リリースしたら**すぐに**顧客からのフィードバック収集やデータ計測を行う
- 高速なPDCAサイクルにより、プロダクトの方向性がズレる前に修正できる

### 部署別の役割分担

| 部署 | 担当内容 |
|------|----------|
| **CS** | ユーザーインタビュー、NPS調査を主導 |
| **BizDev** | 市場性や価格戦略の仮説検証 |
| **エンジニアリング** | フィードバックを踏まえた必要最小限の実装 |

### 検証の仕組み化
- 「リリース後すぐに顧客からフィードバックを収集できる仕組み」を予め準備
  - アプリ内のフィードバック機能
  - オンボーディング時のヒアリング
- **プロダクトリリースと検証活動をワンセット**に計画

### 定期レビュー
- 週次あるいはスプリントごとに各担当が進捗データや顧客の声を共有
- **複数部署のロードマップを付き合わせて調整する場**を設ける
- 組織横断の透明性と協調がロードマップ実行フェーズでは鍵

## 4. 進捗を追う定量KPI・指標の設定と活用

### SaaS/サブスクリプション必須指標

| 指標 | 説明 | 活用ポイント |
|------|------|-------------|
| **ARR/MRR** | 年間/月間経常収益 | 売上成長トレンド、収益基盤の構築状況把握。資金調達時に注目される |
| **チャーン率** | 解約率 | 低いほど顧客満足度が高くスティッキネスが強い |
| **リテンション率** | 継続利用率 | リテンションカーブが**横ばい（安定）**になればPMFに近い |
| **NPS** | ネットプロモータースコア（推奨意向） | 高ければ口コミで広がりやすい。PMF達成度の参考になる |
| **CAC** | 顧客獲得単価 | マーケ効率を示す |
| **LTV** | 顧客生涯価値 | **LTV > CAC**であればビジネスはスケールしやすい |

### リテンションカーブの読み方
- ある時点で**横ばい（安定）**になる → 一定のユーザーに長期価値を提供できている＝**PMFに近い**
- **右肩下がり**に落ち続ける → まだ市場にフィットしていない、離脱理由の分析と対策が必要

### NPSの活用
- 1つの質問で調査でき、小規模組織でも導入しやすい
- 「推奨者を増やし批判者を減らす」という明確な目標設定が可能
- 投資家へのアピール材料としても有効

### KPIモニタリング体制
- OKRやダッシュボードで継続モニタリング
- **月次・四半期ごと**にレビューする体制を整備
- 指標に大きな変動があれば要因を分析し、施策を改善するフィードバックループに繋げる
- 定量指標を**"現在地を示す計器"**として活用 → 主観ではなく客観データに基づいた意思決定

## 5. 低優先度機能の「今はやらない」切り分け方

### 優先順位ランク付け基準

| 優先度 | 内容 | 対応 |
|--------|------|------|
| **Must-have** | 事業戦略上で最重要な事項 | ロードマップに載せる |
| **コア価値** | プロダクトのコア価値に関わるもの（差別化要素） | ロードマップに載せる |
| **課題解決** | ターゲットユーザーの主要な課題を解決するもの | ロードマップに載せる |
| **Nice-to-have** | あれば嬉しい改善（UX向上などの付加価値） | 状況に応じて検討 |
| **Not Now** | 現時点では必須ではない価値（無くても運用できる機能） | バックログでアイデア管理のみ |

### 「Not Nowリスト」の運用
- 下位に位置づけられた「現時点で必須でない」機能は**プロダクトバックログでアイデアとして管理**
- 公式なロードマップには載せない
- 将来の検討材料として「**Not Nowリスト**」や「**アイスボックス**」に入れる
- リソースに余裕ができた段階や市場の状況変化に応じて改めて評価

### リファクタリングの考え方
- ユーザーに直接価値を与えないリファクタリングや技術的刷新は**成長や収益を阻害するボトルネックになった時にのみ着手**
- 「顧客が喜ぶ新機能の開発や収益拡大に不可欠でない大規模リファクタは、スタートアップでは後回しにすべき」（Thomas Tunguz）

### 判断基準
常に以下の観点で問い直す：
> **「それを今やることで顧客がより幸せになるか？収益が伸びるか？」**

答えがNOなら**今はやらない**という判断を徹底する。

### プロダクトマネージャーの役割
- 明確な**優先順位の盾**となり、「それは今やらない」と宣言する勇気を持つ
- ロードマップ策定段階で**スコープから除外する事項**を共有しておく
- 「今は○○を優先するため、その機能は後で検討しましょう」と建設的に応答

## まとめ：ロードマップ策定・実行の鍵

> **「重点志向で動き、指標で検証し、学習しながら素早く舵を切る」**

1. **MVPからPMFまで段階的に進む戦略**（CPF→PSF→SPF→PMF）
2. **OKR/KPIによるチームの目線合わせ**
3. **部署横断の検証サイクル**（Build-Measure-Learn）
4. **データドリブンな改善指標**（ARR/MRR、チャーン率、NPS、LTV/CAC）
5. **大胆な取捨選択**（Not Nowリスト、優先順位の盾）
