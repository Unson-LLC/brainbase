# § 1. EOS経営フレームワーク

**目的**: 中小・スタートアップ経営でビジョン共有・実行力（トラクション）を高める

**Why**:
- 組織を強化し成長を推進
- 経営者が「火消し」から「設計・管理」へシフト
- シンプルかつ効果的なイノベーション実現

**Source Skill**: eos-framework（127行）

---

## 1.1 EOSの基本構造（6モジュール）

EOSは、成功を収めている経営者が実践している、成功するためのシンプルな方法論です。

### 6つのモジュール

1. **ビジョン（Vision）**
2. **人（People）**
3. **データ（Data）**
4. **課題（Issues）**
5. **プロセス（Process）**
6. **トラクション（Traction / 実行力）**

**思考パターン**:
```
組織課題を見たら → 「6モジュールのどれに該当するか？」
ビジョン不明確 → ビジョンモジュール
人が定着しない → 人モジュール
数字が見えない → データモジュール
問題が解決しない → 課題モジュール
属人化 → プロセスモジュール
実行されない → トラクションモジュール
```

---

## 1.2 ビジョン

**目的**: 経営チームと全従業員が同じビジョンを共有し、明確な方向性を理解している状態を作る

| 概念/ツール | 詳細 |
| :--- | :--- |
| **ビジョン・トラクション・シート** | ビジョンを定義、共有するためのシンプルな2ページで構成されるツール |
| **ビジョン8つの質問** | コア・バリュー、コア・フォーカス、10年目標、マーケティング戦略、3年イメージ、1年計画、四半期の石、課題 |
| **コア・バリュー** | 会社にとって不変の基本的な指針や原則（採用、解雇、評価の基準） |
| **コア・フォーカス** | 組織の存在理由、使命、パッションを定義し、ニッチ（得意とする分野）を明確にする |
| **10年目標 / 3年イメージ / 1年計画** | 長期的な方向性を定める大規模な目標と、達成時期を設定したより具体的な中間目標 |
| **マーケティング戦略** | ターゲット市場、3つのユニーク（差別化要因）、証明されたプロセス、満足保証の4要素 |

---

## 1.3 人（正しい人・正しい席）

**目的**: 優秀な人々（「正しい人」）で組織を囲み、「正しい席」に配置する

| 概念/ツール | 詳細 |
| :--- | :--- |
| **正しい人** | 会社のコア・バリューを共有し、実践する人。コア・バリューに合わない人を排除する基準にもなる |
| **正しい席** | 従業員がその役割に対してGWC（業務を理解している、やる気がある、能力がある）の条件を満たす場所 |
| **ピープル・アナライザー** | 従業員がコア・バリューに沿った行動をしているかを評価するツール |
| **アカウンタビリティ・チャート** | 組織の構造図であり、権限と責任を明確にするためのツール（営業/マーケティング、オペレーション、バックオフィスの3機能） |
| **LMA（結果責任者）** | 各機能（席）には、一人のリーダー、マネージャー、結果責任を持つ人が割り当てられる |
| **ビジョナリーとインテグレーター** | ビジョナリー（創業者など）はアイデア・創造性、インテグレーターは指示・管理・結果責任を担う |

**判断パターン**:
```
採用判断 → 「コア・バリューを共有しているか？」
配置転換 → 「GWC（Get it / Want it / Capacity）を満たしているか？」
```

---

## 1.4 データ（スコアカード）

**目的**: 感情や憶測ではなく、数値に基づいて客観的に会社を管理し、冷静さを取り戻す

| 概念/ツール | 詳細 |
| :--- | :--- |
| **スコアカード** | 会社の状態を毎週監視するための5〜15個の主要な先行指標（KPI）を追跡するツール |
| **測定可能な数字** | 全員に数字目標を持たせ、透明性を確保し、責任感を生み出す |

**思考パターン**:
```
経営判断 → 「数値根拠はあるか？」
感覚的な意見 → 「スコアカードで確認」
先行指標の確認 → 「5〜15個のKPIを毎週追跡」
```

---

## 1.5 課題（IDS）

**目的**: 問題や障害を特定し、オープンに議論し、根本原因を解決する

| 概念/ツール | 詳細 |
| :--- | :--- |
| **課題リスト** | 組織内で浮上したあらゆる問題や課題を書き出し、リスト化する |
| **課題解決トラック (IDS)** | 1. **Identify（特定）**: 根本原因を掘り起こす<br>2. **Discuss（議論）**: 根本原因に対する選択肢をオープンに議論<br>3. **Solve（解決）**: 最善の解決策を決定し、次の行動（To Doリスト）を決める |

**思考パターン**:
```
問題発生 → 「根本原因は何か？」（Identify）
根本原因特定 → 「選択肢は？」（Discuss）
選択肢整理 → 「最善の解決策と次の行動は？」（Solve）
```

---

## 1.6 プロセス

**目的**: 会社の「ウェイ（やり方）」を特定し、文書化し、全従業員がそれに従うようにする

| 概念/ツール | 詳細 |
| :--- | :--- |
| **コア・プロセス** | 会社を構成する主要な6〜7のプロセス（人事、営業、オペレーション、会計など）を特定する |
| **コア・プロセスの文書化** | 80/20の法則に基づいて最も重要な20%のステップに焦点を当てて、1〜2ページに文書化する |
| **全員実行** | 文書化されたプロセスを全従業員に教え、遵守するように管理する |

**思考パターン**:
```
属人化発見 → 「コアプロセスを文書化したか？」
プロセス文書化 → 「80/20の法則で1〜2ページにまとめる」
文書化完了 → 「全員に教え、遵守を管理」
```

---

## 1.7 トラクション（実行力）

**目的**: ビジョンと日々の活動を結びつけ、規律と責任をもってアイデアを行動に移す

| 概念/ツール | 詳細 |
| :--- | :--- |
| **石 (Rocks)** | 組織の90日間に達成すべき3〜7つの最優先事項 |
| **ミーティングのリズム** | 定期的なミーティングを確立し、コミュニケーションと生産性を向上させる |
| **四半期ミーティング** | 90日のサイクルの終わりに、経営チームがオフサイトで集まり、前四半期の成果を見直し、次の四半期の「石」を設定する |
| **10点満点ミーティング** | 毎週90分間の定例会議（切り替え、スコアカード、石の進捗、顧客・従業員のニュース、To Doリスト、課題解決トラック (IDS)） |
| **To Doリスト** | 毎週のミーティングで決定された「次の行動」のリスト。各項目に責任者を定め、1週間以内に完了させる |

**思考パターン**:
```
四半期目標 → 「3〜7つの石を設定」
毎週定例 → 「10点満点ミーティング（90分）」
ミーティング → 「切り替え → スコアカード → 石 → ニュース → To Do → IDS」
To Do → 「責任者明確、1週間以内に完了」
```

# § 2. 財務諸表経営判断

**目的**: PL/BSから経営判断を導く思考フレームワークを提供

**Why**:
- 財務数値から課題を発見
- 主要指標の判断基準を明確化
- アクション導出までの体系的アプローチ

**Source Skill**: financial-statement-framework（528行）

---

## 2.1 PL/BSの本質

### PL（損益計算書）= 稼ぐ力を見る

```
売上高
  ↓
－ 売上原価
  ↓
＝ 売上総利益（粗利）    ← 商売の基礎体力
  ↓
－ 販売費及び一般管理費  ← 固定費
  ↓
＝ 営業利益              ← 本業の稼ぐ力
  ↓
± 営業外損益
  ↓
＝ 経常利益              ← 通常の経営活動の結果
  ↓
± 特別損益
  ↓
－ 法人税等
  ↓
＝ 当期純利益            ← 最終的に手元に残る利益
```

**本質**: 「今期どれだけ稼いだか（または損したか）」のフロー

### BS（貸借対照表）= 財務の健全性を見る

```
【資産の部】          【負債の部】
流動資産              流動負債
  現預金                未払金
  売掛金                未払費用
  棚卸資産              短期借入金

固定資産              固定負債
  有形固定資産          長期借入金
  無形固定資産
  投資その他          【純資産の部】
                        資本金
                        利益剰余金
────────────      ────────────
資産合計           ＝  負債＋純資産合計
```

**本質**: 「今、何を持っていて、どう調達したか」のストック

---

## 2.2 PLの読み方（5段階チェック）

### ステップ1：売上総利益率を見る

```
売上総利益率 ＝ 売上総利益 ÷ 売上高 × 100

【判断基準】
✅ 70%以上   → 非常に良好（SaaS、コンサル等）
✅ 50-70%    → 良好（サービス業）
✓ 30-50%    → 普通（製造業、小売業）
⚠ 30%未満   → 薄利（要改善）
```

**思考チェーン**:
- 高い（50%以上）→ サービス業的 → 販管費管理が重要
- 低い（30%未満）→ 仕入れ・外注依存 → 原価削減 or 売上単価UP

### ステップ2：営業利益率を見る

```
営業利益率 ＝ 営業利益 ÷ 売上高 × 100

【判断基準】
✅ 10%以上   → 優良
✓ 5-10%     → 普通
⚠ 0-5%      → ギリギリ
❌ マイナス  → 赤字（本業で損失）
```

### ステップ3：費用構造を分析する

主要費用の洗い出し：
1. 売上高に対する比率を計算
2. 上位5項目を特定
3. 異常値（極端に高い項目）を発見

| 費用項目 | 売上比率 | 判断 | アクション |
|---------|---------|------|-----------|
| 外注費 | 60%超 | ⚠ 依存度高すぎ | 内製化 or 外注先見直し |
| 人件費 | 50%超 | ⚠ 労働集約的 | 生産性向上 or 売上拡大 |
| システム費 | 10%超 | ⚠ IT投資過多 | ツール統廃合 |
| 役員報酬 | 20%超 | ⚠ 経営者依存 | 事業規模拡大が先決 |

### ステップ4：経常利益と当期純利益の差を見る

**パターン**:
- 経常利益 -300万円 → 当期純利益 -50万円
  → 繰越欠損金 or 税制優遇で損失圧縮（ポジティブ）

- 経常利益 +300万円 → 当期純利益 -100万円
  → 特別損失が大きい（要調査）

### ステップ5：損益分岐点を推定する

```
固定費 ＝ 販管費 + その他固定的な費用
変動費率 ＝ 売上原価 ÷ 売上高

損益分岐点売上高 ＝ 固定費 ÷ (1 - 変動費率)
```

**思考チェーン**:
- 現在売上が損益分岐点の90%以上 → あと少しで黒字化
- 現在売上が損益分岐点の70%未満 → 売上拡大 or 構造改革必要

## 2.3 BSの読み方（3段階チェック）

### ステップ1：自己資本比率を見る

```
自己資本比率 ＝ 純資産 ÷ 総資産 × 100

【判断基準】
✅ 50%以上    → 非常に安定（無借金経営可）
✅ 30-50%     → 安定
✓ 10-30%     → 普通（銀行融資は可能）
⚠ 0-10%      → 低い（財務脆弱）
❌ マイナス   → 債務超過（緊急事態）
```

### ステップ2：流動比率を見る

```
流動比率 ＝ 流動資産 ÷ 流動負債 × 100

【判断基準】
✅ 200%以上   → 非常に良好
✅ 150-200%   → 良好
✓ 100-150%   → 普通
⚠ 100%未満   → 短期支払能力に懸念
```

### ステップ3：現預金残高を確認する

```
月商 ＝ 年間売上 ÷ 12

現預金月数 ＝ 現預金残高 ÷ 月商

【判断基準】
✅ 6ヶ月分以上  → 非常に安心
✅ 3-6ヶ月分   → 安心
✓ 1-3ヶ月分   → 普通
⚠ 1ヶ月分未満  → 危険
```

## 2.4 課題発見とアクション導出

### 発見パターン1：売上は伸びているが利益が出ない

**症状**:
- 売上高：前年比+20%
- 営業利益：マイナス継続

**診断**:
1. 売上総利益率を確認 → 低い場合は原価管理の問題
2. 費用の伸び率を確認 → 売上以上に伸びていれば固定費肥大化
3. 外注費・人件費の比率を確認

**アクション**:
□ 原価率の高い案件を見直し
□ 固定費の削減（不要なツール・契約の解約）
□ 高粗利案件の選別強化

### 発見パターン2：黒字だが資金繰りが苦しい

**症状**:
- 営業利益：プラス
- 現預金：少ない
- 流動比率：100%未満

**診断**:
1. 売掛金が多い → 回収サイトが長い
2. 在庫が多い → キャッシュが在庫に化けている
3. 借入返済が多い → BSの長期借入金を確認

**アクション**:
□ 入金サイクルの短縮交渉
□ 前払い・中間金の導入
□ 在庫圧縮
□ 借入返済の見直し（リスケ検討）

## 2.5 判断の型：3つの経営レバー

### レバー1：売上を増やす

**いつ使うか**:
- 損益分岐点に近い（達成率90%以上）
- 粗利率が高い（50%以上）
- 現預金に余裕がある（3ヶ月分以上）

**アクション**:
- 営業強化
- 新規顧客開拓
- 既存顧客の単価UP・リピート促進

### レバー2：コストを減らす

**いつ使うか**:
- 売上拡大が困難
- 特定費用が異常に高い（売上比率で判断）
- 損益分岐点が遠い（達成率70%未満）

**アクション**:
- 外注費・人件費の最適化
- 固定費の削減（ツール統廃合、賃料見直し）
- 低収益案件の撤退

### レバー3：資金繰りを改善する

**いつ使うか**:
- 流動比率100%未満
- 現預金1ヶ月分未満
- 債務超過

**アクション**:
- 入金サイクル短縮
- 支払サイト延長交渉
- 資金調達（融資・増資）
- 不要資産の売却

## 2.6 定期レビューのルーチン

### 月次レビュー（15分）

```
1. PL速報値チェック
   □ 売上：予算比 ±10%以内か
   □ 粗利率：前月比で大きな変動ないか
   □ 営業利益：予算比でプラスか

2. 資金繰りチェック
   □ 現預金残高：前月比で減少していないか
   □ 今月の大きな支払予定：問題なく払えるか

3. アクション
   □ 異常値があれば原因を調査
   □ 対策を即座に実行
```

```
2. BS健全性チェック

   □ 流動比率：100%以上を維持できているか

3. 戦略見直し
   □ 現在の戦略（売上拡大 or コスト削減）は正しいか
   □ 目標達成のペースは適切か
```

# § 3. システム思考

**目的**: 業務・人生をシステムとして分解/文書化し、ムダな火消しを減らして継続的改善を回す

**Why**:
- 属人化した業務を手順化・標準化
- 問題対応が後手で火消しが多い状況を改善
- 視点を"outside and slightly elevated"に切り替え

**Source Skill**: work-the-system（78行）

---

## 3.1 システム思考の世界観

### 現実の機械的性質

- **世界はシステムで構成されている**: あなたの人生は、個人的、専門的、財政的、社会的、生物学的、機械的なシステムのコレクション
- **システムの普遍性**: 現実の世界は、99.99%が完璧に機能しているシステムに基づいている
- **混沌（カオス）の排除**: 多くの人が日常をアモルファス（不定形）で複雑な事柄の寄せ集めだと認識しているが、システム思考を持つと、世界がシンプルで予測可能であることに気づく

**思考パターン**:
```
混沌に見える状況 → 「これはシステムの集合体」
問題発生 → 「システムの機械的な欠陥」
運や宿命のせい → 「システムの不備」
```

---

## 3.2 視点の変更（Outside and Slightly Elevated）

### 一歩引いた視点

自分のシステムの働きを客観的に観察するために、自分の世界を**外部から、わずかに高い視点（Outside and slightly elevated / One layer deeper）**から見る必要があります。

**思考パターン**:
```
火消しに追われる → 「一歩引いて俯瞰」
日々の出来事 → 「裏にある機械的なプロセスを理解」
コントロール喪失 → 「システムを調整・作成・削除する能力を持つ」
```

---

## 3.3 3つのステップ（分離・文書化・維持）

### 1. 分離、解剖、システムの修復

- **問題システムの特定**: まず、あなたの世界を構成する個々のシステムを分離し、分析
- **非効率性の排除**: 望ましくない結果を生み出しているシステムを修復または排除
- **システム改善（System Improvement）**: 問題を「改善のための赤い旗（Red Flag for improvement）」と見なす

### 2. 文書化

- **目標、原則、手順の作成**: 改善されたシステムを実行するための書面による目標、一般的な運営原則、作業手順を作成
- **「防火環境」の構築**: 文書化は、繰り返しの問題による損失を防ぐための「防火環境」を構築する「一度限りの重労働」

### 3. 継続的な維持

- システムが最高の効率で稼働し続けるように、継続的に監視、調整、および維持を行う

**思考パターン**:
```
問題発見 → 「システムを分離・解剖・修復」
修復完了 → 「文書化して再発防止」
文書化完了 → 「継続的に維持・改善」
```

---

## 3.4 3つの主要文書

| 文書 | 目的と特徴 |
| :--- | :--- |
| **1. 戦略的目標 (Strategic Objective)** | 会社の全体的な方向性、ターゲット、および戦略を定義する1ページの簡潔な文書 |
| **2. 一般的な運営原則 (General Operating Principles)** | 意思決定のための指針（ガイドライン）となる、簡潔な原則のコレクション（2〜3ページ程度） |
| **3. 作業手順 (Working Procedures)** | 個々のシステム（職場のタスク）の実行方法を正確に記述した文書。1-2-3-4ステップの線形フォーマット |

**思考パターン**:
```
組織の方向性 → 「戦略的目標（1ページ）」
意思決定の判断基準 → 「一般的な運営原則（2〜3ページ）」
具体的なタスク → 「作業手順（1-2-3-4ステップ）」
```

---

## 3.5 実践的なシステム改善戦略

### 1. 実行力と規律

- **怠慢の過ち (Errors of Omission) の克服**: 意図的に行動を起こさないこと（先延ばしや怠惰）は、人生の目標達成を妨げる主要な原因
- **「今すぐやる」（Point-of-Sale Thinking）**: タスクを自動化 (Automate)、委任 (Delegate)、または破棄 (Discard)
- **ゆっくりと、しかし確実に**: 急いで結果を出そうとするのではなく、システムの継続的な改善を通じて、着実に進歩

### 2. 時間とエネルギーの最適化

- **生物学的プライムタイム（BPT）**: 自分の生物学的構成に基づいて、精神的なエネルギーがピークに達する時間帯を特定し、その時間を最も重要で創造的なタスクに割り当てる
- **機械的プライムタイム（MPT）**: 実際に商品やサービスを生み出すことに費やされる時間
- **「より少なく働く」**: システム思考により、仕事量を週40時間以下に削減

**思考パターン**:
```
先延ばし発見 → 「自動化・委任・破棄のどれか」
重要タスク → 「BPT（生物学的プライムタイム）に割り当て」
過労 → 「システム改善で週40時間以下に削減」
```

# § 4. タスク委譲SL理論

**目的**: 部下の習熟度に応じてタスク委譲方法を最適化

**Why**:
- 同じ人でもタスクによってレベルは変わる
- 適切な委譲方法で部下の成長を促進
- マイクロマネジメントと放任の適切なバランス

**Source Skill**: task-delegation-sl（147行）

---

## 4.1 習熟度の2軸（能力×意欲）

| 軸 | 内容 |
|---|------|
| **能力（Ability）** | スキル・知識・経験 |
| **意欲（Willingness）** | モチベーション・自信・責任感 |

**重要**: 同じ人でもタスクによってレベルは変わる。「人のレベル」ではなく「人×タスクのレベル」で判断。

**思考パターン**:
```
タスク委譲 → 「この人はこのタスクに対して能力があるか？」
能力確認 → 「意欲はあるか？」
能力×意欲 → 「M1/M2/M3/M4を判定」
```

---

## 4.2 4段階の習熟度×リーダーシップスタイル

| レベル | 能力 | 意欲 | スタイル | 指示量 | サポート量 |
|--------|------|------|----------|--------|------------|
| **M1** | 低 | 高/低 | **教示型（Telling）** | 高 | 低 |
| **M2** | 低〜中 | 低 | **説得型（Selling）** | 高 | 高 |
| **M3** | 高 | 変動 | **参加型（Participating）** | 低 | 高 |
| **M4** | 高 | 高 | **委任型（Delegating）** | 低 | 低 |

---

## 4.3 レベル別チェックリスト

### 全レベル共通（必須）

```
□ 責任者は1人か（LMA明確化）
□ 期限は設定したか
□ 成功/失敗の基準を伝えたか
□ 必要なリソース（情報・ツール・人）を渡したか
□ 報告ライン（誰に報告するか）は明確か
```

### M1：新人・未経験者（能力低・意欲不安定）

| # | 必須項目 |
|---|---------|
| 1 | **手順を1-2-3で具体的に指示**したか |
| 2 | **成果物のサンプル/テンプレート**を渡したか |
| 3 | **細かい期限**（1日単位）を設定したか |
| 4 | **毎日の進捗確認**を設定したか |
| 5 | 質問しやすい環境を作ったか |
| 6 | 失敗しても致命傷にならないタスクを選んだか |

**ポイント**: 「何を」「どうやって」を細かく指示

### M2：成長中（能力発展中・意欲低下リスク）

| # | 必須項目 |
|---|---------|
| 1 | **なぜやるか（目的）**を説明したか |
| 2 | 本人に**計画を立てさせ**たか |
| 3 | 計画への**フィードバック**をしたか |
| 4 | **週1-2回**の進捗確認を設定したか |
| 5 | **良い点を先に伝えて**からアドバイスしたか |
| 6 | 本人の**工夫・意見**を聞いたか |

**ポイント**: 「なぜ」を伝え、双方向コミュニケーション

### M3：中堅・一人でできるが不安あり（能力高・意欲変動）

| # | 必須項目 |
|---|---------|
| 1 | **ゴール（期待する結果）**だけ伝えたか |
| 2 | **やり方は本人に任せ**たか |
| 3 | **判断基準・NGライン**を伝えたか |
| 4 | **相談先**を明示したか |
| 5 | **週1回程度**の確認で十分か判断したか |
| 6 | 本人の**意思決定を尊重**したか |

**ポイント**: 「何を達成するか」を伝え、やり方は任せる

### M4：ベテラン・自走可能（能力高・意欲高）

| # | 必須項目 |
|---|---------|
| 1 | **目的と期限**だけ伝えたか |
| 2 | **権限を明確に委譲**したか |
| 3 | **報告頻度は本人に任せ**たか |
| 4 | **マイクロマネジメントを避け**たか |
| 5 | 必要なときだけ**サポート提供**する体制か |
| 6 | 完了後に**認知・感謝**を伝えたか |

**ポイント**: 「任せる」。介入は最小限

---

## 4.4 よくある失敗パターン

| 失敗 | 原因 | 対策 |
|------|------|------|
| 「思ったよりやってくれなかった」 | 説明がざっくりすぎ | レベル判定→M1/M2なら具体指示 |
| 部下が動けない | 責任だけ渡して権限なし | 権限と責任をセットで委譲 |
| 納期遅延 | 優先順位・判断基準を伝えていない | NGライン・優先度を明示 |
| 部下が萎縮 | マイクロマネジメント（過干渉） | M3/M4なら介入を減らす |
| 部下の成長が止まる | いつまでもM1対応 | 成長に合わせてスタイル変更 |

---

## 4.5 委譲の4P

| P | 内容 |
|---|------|
| **Purpose（目的）** | なぜこのタスクを委譲するのか |
| **Priorities（優先度）** | どのタスクを先にやるべきか |
| **People（人）** | 適切なスキル・レベルの人を選ぶ |
| **Process（プロセス）** | レベルに応じた手順・やり方を示す |

**思考パターン**:
```
タスク委譲前 → 「4P（Purpose/Priorities/People/Process）を確認」
Purpose明確化 → Priorities設定 → People選定 → Process決定
```

# § 5. レベニュー組織設計

**目的**: B2Bのリード〜商談〜受注〜CSまでのハンドオフ基準を整備し、レベニュー組織のボトルネック診断とKPI設計を行う

**Why**:
- マーケ・インサイドセールス・営業・CSの共業プロセスを再現性高く設計
- 組織の構造化、プロセスの管理、成果の最大化

**Source Skill**: the-model-revenue-process（109行）

---

## 5.1 ザ・モデルの核となる概念

### 1. ビジネスにおける「再現性」の追求

ビジネスにおいて最も重要なことの一つは「再現性」であり、この再現性を実現するための体系的なプロセス管理手法として「ザ・モデル」を活用します。

### 2. 営業プロセスの分業から共業へ

従来の営業プロセスは、一人の営業担当者がすべてをカバーするものでしたが、「ザ・モデル」では、営業プロセスを専門機能ごとに**分業**し、部門間の連携（**共業**）を重視します。

| 部門 | 主な役割 |
| :--- | :--- |
| **マーケティング** | リード獲得と認知拡大。ウェブサイト、キャンペーン、セミナーなどを通じた活動量の最大化 |
| **インサイドセールス** | リードの育成（ナーチャリング）と、商談への移行（クオリファイ）を行う商談供給の調整弁 |
| **営業 (AE/フィールドセールス)** | 有望リードに対する提案、交渉、契約プロセスを管理し、受注を獲得する |
| **カスタマーサクセス (CS)** | 顧客の活用促進、リテンション、アップセル/クロスセルを支援する |

---

## 5.2 リードと商談のプロセス管理

### 1. リードの段階的評価（顧客ステージ）

**顧客ステージ（マーケティング/インサイドセールス担当範囲）**:

1. **認知拡大**: ウェブサイト訪問数、イベント参加者数など
2. **リード獲得**: メールアドレスなどのコンタクト情報の取得
3. **リード育成**: メールリンクのクリックや製品サイトへのアクセス、セミナー参加など
4. **有望リード**: 商品カタログ請求、問い合わせ、またはリードスコアが100点以上など

### 2. リサイクル（Recycle）の重要性

新規リードの約65%は「将来購入の可能性はあるが今すぐではない」という見込み客であり、ナーチャリング（継続的フォロー）が必要です。

- **失注・未商談リードの価値**: 受注・未商談リードは、コストをかけずに獲得できる商談機会の「宝の山」
- **プロセスの再構築**: マーケティングオートメーション（MA）を活用した再商談化（リサイクル）の流れを作る

### 3. 営業のフェーズ管理

| フェーズ | 確度 | 定義 (エッセンス) | 移行判定基準 (例) |
| :--- | :--- | :--- | :--- |
| **フェーズ 1** | 0% | リード以上、商談未満の状態 | 自社製品やサービスの解決可能性、商談未満の状態でいること |
| **フェーズ 2** | 25% | 顧客のビジネス課題を認識し、自社の解決手段を理解する | BANT条件（予算、決裁権、必要性、導入時期）を確認できたこと |
| **フェーズ 3** | 30% | 評価と選定 | 競合他社との差別化や優位性に基づいて、顧客に選択条件を自ら作り出すことが必要 |
| **フェーズ 4** | 50% | 最終交渉と意思決定 | 正式に稟議プロセスを開始し、契約を締結するための道筋が整うこと |
| **フェーズ 5** | 75% | 稟議決裁プロセス | Mutual Close Plan (顧客と相互に合意した計画) の完了 |

---

## 5.3 三つの基本戦略

### 1. 市場戦略（選択と集中）

| 項目 | SMB市場 | エンタープライズ市場 |
| :--- | :--- | :--- |
| **市場戦略のメリット** | 安定したビジネス基盤 | アップサイドを狙う成長基盤 |
| **セールス** | Volume x Velocity（数と回転率）| Value（金額や価値） |
| **人材タイプ** | 規律・オペレーション（左脳型）| 独創性・コンサルテーション（右脳型） |

### 2. リソースマネジメント

- **事業の因数分解**: 売上を「直販/代理店」や「新規顧客/既存顧客」に分解し、さらに「商談件数」「受注率」「案件単価」などの指標に分解
- **セールスキャパシティ**: 営業一人あたりの生産性を正確に計算し、目標達成に必要な人員数を確保
- **生産性のレバー**: 見込客数、受注率、案件単価を上げるか、営業人数、商談期間を下げる

### 3. パフォーマンスマネジメント

- **トレンドの重視**: 絶対値ではなく、相対的なトレンド（変動）を見て管理
- **複数指標による分析**: 受注率や商談化率を「件数ベース」と「金額ベース」の2つで分析
- **ボトルネックの特定**: 組織全体の生産性を妨げているボトルネックは一つだけ

---

## 5.4 人材・組織・リーダーシップ

### 1. 組織作りのベストプラクティス

- **バリューの共有**: 組織全体でビジョン、ミッション、バリューを共有
- **多様性**: 多様な個性を持つメンバーが共通の価値観を共有
- **採用基準**: 「一緒に働きたいと思えるかどうか」「自社のバリューを共有するか」

### 2. リーダーシップのスタイル

- **増幅型リーダーと消耗型リーダー**: メンバーの能力を引き出し、組織の成果を高めるのが「増幅型リーダー」
- **ストレッチゴールの設定**: 達成が困難だが挑戦しがいのあるゴールを設定
- **「利益と尊敬と少しの恐怖」**: リーダーは部下に対して、尊敬の念を持って接するとともに、時にはプレッシャーを与える

# § 6. Skills思考パラダイム

**目的**: Anthropic Custom Skillsの本質的理解を提供し、「思考のフレームワーク」としてのSkills設計原則を定義

**Why**:
- Skillsは「知識提供」ではなく「思考パターン提供」
- AI Agentのpersonality/behaviorを定義
- 一貫した判断基準・意思決定フレームワークを付与

**Source Skill**: skills-paradigm（388行）

---

## 6.1 根本的な誤解と正しい理解

### ❌ 従来の誤解: Skillsは知識のデータソース

多くの開発者が陥る誤解：

```
Skills = 情報検索ツール
Skills = ドキュメントアクセス
Skills = データベース
```

**問題点**:
- LLMはすでに知識を持っている
- 単なる情報提供ではSkillsの真価を発揮できない
- コンテキストウィンドウの無駄遣い

### ✅ 正しい理解: Skillsは思考のフレームワーク

Skillsの本質：

```
Skills = 思考パターン
Skills = 認識のフレームワーク
Skills = AIのpersonality/behavior
```

**効果**:
- LLMに「〇〇ならこう考える」という行動パターンを提供
- 一貫した判断基準・意思決定フレームワークを付与
- プロンプトよりも強力で安定した制御

## 6.2 Skills vs カスタムコマンドの階層構造

### アーキテクチャ上の位置づけ

```
┌─────────────────────────────────────┐
│  カスタムコマンド                    │
│  (/command で明示的実行)             │
│  ・ユーザー操作                      │
│  ・ワークフロー                      │
│  ・内部でSkillsを参照                │
└────────────┬────────────────────────┘
             │ 活用
             ▼
┌─────────────────────────────────────┐
│  Skills                             │
│  (Claudeが自動参照)                  │
│  ・知識・パターン                    │
│  ・ベストプラクティス                │
│  ・思考フレームワーク                │
└─────────────────────────────────────┘
```

### 原則

**Skillsが先に必要**。カスタムコマンドはSkillsを活用する上位レイヤー。

- Skills = 基礎知識・思考パターン（テンプレート、ロジック、ベストプラクティス）
- カスタムコマンド = ユーザー操作・ワークフロー（/commandで明示的実行）

## 6.4 Skills設計原則

### 1. "What"ではなく"How to think"を記述

**NG**: 「タスクフォーマットはYAML front matterです」
**OK**: 「タスクを見たら、まずRACIを確認し、優先度を判断する」

### 2. 判断基準を明確化

**NG**: 「良いタスクを書いてください」
**OK**: 「タスクは動詞で開始、担当者明記、期限設定の3条件を満たす」

### 3. 行動パターンを例示

**NG**: 「RACIを使いましょう」
**OK**: 「責任が曖昧なタスクを見つけたら、RACI形式で関係者を整理し、Responsible担当者を1名に絞る」

### 4. 思考の連鎖を設計

**NG**: 「戦略を考えてください」
**OK**:
```
1. まずICP（理想顧客像）を特定
2. ICPの課題をN=1で深掘り
3. 課題を解決する価値提案を定義
4. 競合との差別化要因を明確化
5. Go-to-Market戦略を立案
```

## 6.5 Orchestration（複数Skillsの連携）

### 単一Skillの限界

1つのSkillでは複雑なワークフローを表現できない。

### Orchestrator Skillパターン

**構造**:
```
{skill}/
├── SKILL.md (メインSkill定義)
└── agents/ (サブエージェント定義)
    ├── phase1.md
    ├── phase2.md
    └── phase3.md
```

**利点**:
1. **コンテキスト効率化**: 各サブエージェントが独自コンテキスト保持
2. **専門化**: 各Phaseに最適なSkillsを選択（8個制限内）
3. **並列実行**: 依存関係のないPhaseは並行実行可能
4. **再利用性**: サブエージェントは他のワークフローでも使用可能

# § 7. プロダクトマネジメント実践

**目的**: PM業務の実践的な判断基準とCOREスキルを提供

**Why**:
- ステークホルダーとの関わり方を最適化
- ユーザーリサーチの実践方法を明確化
- 優先順位付けとロードマップ作成の判断基準を提供

**Source Skill**: pm-shigoto-practice（337行）
**出典**: Product Management in Practice, 2nd Edition（Matt LeMay著）

---

## 7.1 PMの3つの責任

1. **チームメンバーのベストを引き出す**
2. **直属チーム以外の人と一緒に働く**
3. **曖昧さに対処する**

> 「何かがわからない状況を把握するスキルは、その後で何かをするスキルと同じくらい重要」

### 悪いPMのパターン

1. **ジャーゴンジョッキー** - 専門用語で煙に巻く
2. **スティーブ・ジョブズ信奉者** - ビジョナリーを気取りユーザーの声を無視
3. **英雄のPM** - 自分だけで全部やろうとする
4. **張り屋さん** - 達成より働いた時間を誇示
5. **殉教者** - 責任転嫁と言い訳に終始

**思考パターン**:
```
意思決定時 → 「英雄のPMになっていないか？チームを巻き込んでいるか？」
専門用語使用時 → 「ジャーゴンで煙に巻いていないか？明快か？」
```

---

## 7.2 COREスキル

### 1. コミュニケーション

**核心**: 明快さと心地よさのバランス

**自己チェック**:
- やりにくい会話をファシリテーションしているか？
- 遠回しではなく単刀直入に伝えているか？
- 相手の発言に熟慮深く反応しているか？

**Disagree & Commit**:
1. 異議を表明する（明確に）
2. 理由を説明する
3. 決定が下されたらコミットする
4. 決定後は全力で推進する

**NG行動**:
- 陰で不満を言う
- 決定後に足を引っ張る
- 「言ったとおりになった」と後出し

### 2. 組織化

**核心**: 自らを不要とせよ

**自己チェック**:
- 1週間休んでも優先順位付けとデリバリーができるか？
- 「何をしているのか？なぜ？」に誰でも答えられるか？
- チームに必要な情報に簡単にたどり着けるか？

### 3. リサーチ

**核心**: 好奇心をもって生きよ

**自己チェック**:
- 週1回以上ユーザーと話しているか？
- 意思決定はビジネスゴールとユーザーニーズに基づいているか？
- ユーザーニーズは実際のユーザーの声を反映しているか？

### 4. 実行

**核心**: すべての努力はアウトカムのために

**自己チェック**:
- ビジネスインパクトへの複数の道筋を持っているか？
- 戦略的ゴールがスプリント計画の中心になっているか？
- 作業量の積み上げで正当化していないか？

**思考パターン**:
```
スプリント計画 → 「戦略的ゴールが中心か？作業量の積み上げになっていないか？」
ユーザーリサーチ → 「週1回以上ユーザーと話しているか？」
組織化 → 「1週間休んでもチームは回るか？」
```

---

## 7.3 ステークホルダー管理

### 「権力」から情報へ

- 上位者の発言を鵜呑みにしない
- 「なぜ」を理解する
- 情報として受け止め、分析する

### 「いいね、やって」の罠

上位者が安易にGOを出すと、チームが混乱する。

**対処法**:
1. 本当に優先すべきか確認
2. 既存の計画との整合性を問う
3. 何を止めるべきか明確にする

### 上にマネジメントする

- 上司の成功を助ける
- 上司の上司の視点を理解する
- 相手の関心事に寄り添う

**思考パターン**:
```
上位者の指示 → 「本当に優先すべきか？既存計画との整合性は？」
新規依頼 → 「何を止めるべきか明確にする」
```

---

## 7.4 ユーザーリサーチ

### ステークホルダー ≠ ユーザー

| ステークホルダー | ユーザー |
|------------------|----------|
| 社内の関係者 | 実際にプロダクトを使う人 |
| ビジネス目標を持つ | 自分の課題を解決したい |
| 発言力がある | 声が届きにくいことも |

### リサーチのベストプラクティス

1. **週1回以上**ユーザーと話す
2. **行動を観察**する（言葉だけでなく）
3. **オープンな質問**をする
4. **仮説を持ちつつ**修正する準備をする

**ペルソナの罠**: 架空のペルソナより実在のユーザー。「典型的なユーザー」は存在しない。

**思考パターン**:
```
ユーザーリサーチ → 「週1回以上話しているか？」
ペルソナ作成 → 「実在のユーザーベースか？架空のペルソナになっていないか？」
```

---

## 7.5 アジャイル実践

### 3つの迷信

1. **アジャイル＝スクラム** → 方法論は手段
2. **アジャイル＝計画なし** → 適応的計画は計画
3. **アジャイル＝速い** → 学習を重視

### マニフェストからモンスターへ

形式的なアジャイルは本末転倒。「アジャイルをやる」より「アジャイルである」。

## 7.6 ドキュメントとロードマップ

### 「俺たちは要件の奴隷だ！」

仕様書はプロダクトではない。過剰なドキュメントは害になる。

### 安全なドキュメント

- **Living Document**: 常に更新される
- **SSOT（Single Source of Truth）**: 唯一の真実の情報源
- **最初のドラフトは1ページ**: 詳細は後から

### ロードマップの原則

1. **方向性を示す**（詳細な計画ではない）
2. **コミットメントではない**（可能性を示す）
3. **定期的に見直す**

**思考パターン**:
```
ドキュメント作成 → 「Living Documentか？SSOTか？最初は1ページか？」
ロードマップ → 「方向性を示すだけ。コミットメントではない」
```

---

## 7.7 優先順位付け

### すべてを食べることはできない

- 「全部やる」は何もやらないと同じ
- トレードオフを明確に
- 「No」を言う勇気

### フレームワーク

```
RICE: Reach × Impact × Confidence / Effort
ICE: Impact × Confidence × Ease
MoSCoW: Must/Should/Could/Won't
```

### 考え方

- **夢は大きく、スタートは小さく**
- **可逆的な決定**は素早く
- **不可逆的な決定**は慎重に

**思考パターン**:
```
優先順位付け → 「全部やろうとしていないか？トレードオフは明確か？」
決定種類判定 → 「可逆的決定は素早く、不可逆的決定は慎重に」
```

# § 8. AIファースト組織設計

**目的**: 少人数で高生産性なAIファースト組織を設計し、マネジメント・アズ・コード（MaC）を導入するロードマップを提供

**Why**:
- 人をボトルネックにしない組織構造の実現
- 業務の80%をAIが実行する体制の構築
- エンジニアリング中心主義で継続的改善を実現

**Source Skill**: ai-first-org-strategy（96行）

## 8.1 業務主体性の逆転（AI実行率80%）

### 従来型 vs AIファースト

| 項目 | 従来型企業 | AIファースト企業 |
|------|-----------|----------------|
| 業務実行主体 | 人間80% / AI20% | AI80% / 人間20% |
| 一人当たり年間収益 | ~400万円 | ~1,800万円（50倍） |
| 組織のボトルネック | 実行速度・調整コスト | 戦略的判断の質・創造性 |

### Midjourneyの事例

- **チーム規模**: 11人
- **年間収益**: 約300億円（$200M）
- **一人当たり収益**: 約27億円

**業務配分**:
- コンテンツ生成: 100% AI
- カスタマーサポート: 90% AI
- マーケティング: 85% AI

**思考パターン**:
```
組織設計 → 「業務の80%をAIに任せられるか？」
人間の役割 → 「戦略立案・モデル改善・創造性に集中しているか？」
```

## 8.2 エンジニアリング中心主義（60%人材配置ルール）

### 組織構成の逆転

| 役割 | 従来型企業 | AIファースト企業 |
|------|-----------|----------------|
| エンジニア（AI/ML、バックエンド、フロントエンド） | 30% | **60%** |
| プロダクト（PM、デザイナー） | 30% | 20% |
| ビジネス（営業、マーケ、財務、管理） | 40% | 20% |

### Why 60%？

エンジニアリング比率60%が、AI実行80%を可能にする構造的要請：
- 高度な自動化の維持・向上
- 技術的負債の迅速な解消
- AIモデルの継続的改善

**思考パターン**:
```
採用計画 → 「エンジニア比率60%を目指しているか？」
組織設計 → 「仕組み化能力を組織の中核に位置づけているか？」
```

## 8.3 マネジメント・アズ・コード（MaC）

### コンセプト

マネージャーや中間管理職を置かず、従来の管理機能をLLMエージェントやコードベースのガバナンスシステムに代替させる。

### MaCの4つの柱

1. **LLM Chief of Staff**
2. **ルールベース・ガバナンス**
3. **実行可能な知識の埋め込み**
4. **高度なデータ構造化**

---

## 8.4 LLM Chief of Staff

### 機能

LLMが実質的に「AI Chief of Staff (CoS)」として機能：
- プロジェクトの構造化
- タスク分解（高レベル目標 → フェーズ/タスク）
- タイムライン提案
- ボトルネック特定

### 例：新規パートナープログラム立ち上げ

**Input**: 「新しいパートナープログラムを立ち上げたい」

**LLM CoS Output**:
- 必要なタスク一覧
- 責任者割り当て
- 推定期間
- 依存関係マップ

### Cursorのルールベース・ガバナンス

**階層型ルールシステム**:
1. **Team Rules（組織全体）** - 最優先
2. **Project Rules（プロジェクト固有）**
3. **User Rules（個人設定）**

**強制適用機能**:
- Team Rulesで「Enforce this rule」を有効化
- 全チームメンバーに必須
- ユーザーが無効化不可

**効果**:
- 人間の介入なしにコンプライアンス保証
- コードの安全性・スタイルの自動保証
- 人間がボトルネックとなる要因を除去

**思考パターン**:
```
ガバナンス設計 → 「人間の管理をコードベースのルールに代替できるか？」
ポリシー適用 → 「強制適用で自動保証できるか？」
```

## 8.5 既存組織の変革ロードマップ

### Phase 1: エンジニアリング比率の強化

**目標**: エンジニアリング比率を60%程度まで引き上げ

**アクション**:
1. 戦略的な投資と人材配置
2. エンジニアリング採用の優先度最大化
3. 内製化の推進（外注依存の削減）

### Phase 2: MaCの段階的導入

**Step 1: LLM CoS機能の試行**
- 会議メモ・Slack会話の構造化
- タスク明確化の自動化

**Step 2: Cursor型ルールシステムの導入**
- 技術標準・コンプライアンス・セキュリティをルール化
- 人間の介入を排除
- 適用範囲を徐々に拡大

### Phase 3: 透明性と検証可能性の確保

**成功要因**:
- ルールが明確かつ検証可能
- 強制適用はクリティカルな領域に限定
- 技術的指針は柔軟性を持たせる

**思考パターン**:
```
変革計画 → 「Phase 1（エンジニア60%）→ Phase 2（MaC導入）→ Phase 3（透明性確保）」
ルール設計 → 「強制適用はコンプライアンス・セキュリティに限定、他は柔軟に」
```
