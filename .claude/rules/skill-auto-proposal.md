# Skill自動提案ルール

作業中に「これSkill化したほうがいいんじゃない？」と気づいたら、積極的に提案する。

---

## トリガー条件（こういう時に提案を検討）

1. **繰り返しパターンを検知**
   - 似たような手順を複数回実行している
   - 「これ前もやったな」と感じる作業
   - 同じ種類のタスクが頻出している

2. **暗黙知の発見**
   - ユーザーが「いつもこうやってる」と言った
   - 特定の判断基準やワークフローがある
   - 「これ知らないと詰まるやつ」が見つかった

3. **失敗パターンの形式知化**
   - 同じミスを繰り返しそうな箇所
   - 「次は気をつける」で終わりそうな学習
   - gotcha（罠）やtips

---

## brainbase的判断基準（Skill化すべきか？）

### ✓ Skill化すべき

| 基準 | 理由 |
|------|------|
| **循環を加速する** | 詰まりを解消し、出荷量を増やす |
| **正本化に貢献する** | 知識の型化・再現性を高める |
| **アンビエント化に繋がる** | 人の介入を減らし、自動循環を促進 |
| **学習の型化** | 失敗パターンを形式知にして再発防止 |
| **引き継ぎを容易にする** | 属人性を溶かす |

### ✗ Skill化すべきでない

| 基準 | 理由 |
|------|------|
| **一時的な作業** | プロジェクト固有の対応、その場限り |
| **正本を曖昧にする** | 二重管理になる、参照先が増える |
| **例外を増やす** | 運用が複雑になる、型から逸脱 |
| **出荷に繋がらない** | 生成だけの自動化、ドラフト止まり |
| **既存Skillと重複** | 別の場所に同じ知識が増える |

## 提案フロー

### Step 1: 気づきの言語化

作業中に上記トリガーを検知したら、まず内部で確認：

```
- これは繰り返しパターンか？
- brainbaseの循環（Decision→Work→Ship→Learn）のどこを加速するか？
- 既存のSkillsと被ってないか？
```

### Step 2: 既存Skillsチェック

`.claude/skills/` を確認して重複がないか確認。
類似Skillがあれば、そこに追記するほうが良い場合もある。

### Step 3: 価値判断

以下を満たすか確認：

- [ ] 再利用される可能性が高い（3回以上使いそう）
- [ ] 型化することで品質が安定する
- [ ] 別の人（またはAI）が再現できる

### Step 4: 提案

ユーザーに以下の形式で提案：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━
💡 Skill化提案
━━━━━━━━━━━━━━━━━━━━━━━━━━━

【検知したパターン】
[何を繰り返しているか]

【Skill化の価値】
- 循環貢献: [どのLoop段階を加速するか]
- 再利用性: [何回くらい使いそうか]
- 属人性解消: [誰でも再現できるか]

【提案するSkill名】
[skill-name]

【Skill化する？】
- 承認 → Skill作成に進む
- 却下 → 今回は見送り
- 保留 → learning_queueに追加して後で検討
```

---

## 却下すべきケース

以下の場合は提案しない（または「今回は見送り」と判断）：

1. **このセッション限りの作業**
   - 特定のバグ修正、1回きりのデータ移行など

2. **既存Skillの範囲内**
   - 既存Skillを使えば解決する場合

3. **正本が曖昧になる**
   - `_codex/` と `.claude/skills/` で情報が分散する

4. **型化より柔軟性が必要**
   - 毎回判断が異なる、コンテキスト依存が大きい

---

## 承認後のアクション

ユーザーが承認したら：

1. `.claude/skills/{skill-name}/SKILL.md` を作成
2. 既存の `skills-catalog.md` に追加
3. 必要に応じて `/learn-skills` で learning_queue から移動

---

## このルールの適用タイミング

- セッション中、繰り返しパターンに気づいたら**自発的に**提案
- ユーザーに聞かれなくても、価値があると判断したら提案
- ただし作業の流れを止めすぎない（タスク完了後に提案など）
