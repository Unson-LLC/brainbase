## 序文

雲孫合同会社CEOの佐藤です。Tech Knight CTO、SalesTailor CTO、BAAO理事も兼任しながら、1人で4事業を並行運営しています。

先日、BASE柳川さんの「AIに働かされるが正解？」というnoteを読んで、膝を打ちました。特に「ロングコンテキスト問題」という指摘。これ、まさに僕が2年間ずっと格闘してきた問題そのものです。

柳川さんの記事は「組織図がどう変わるか」という視点で整理されていました。素晴らしい抽象化です。

一方、僕は少し違うアプローチを取っています。**「ロングコンテキスト問題を解決するための基盤を先に作る」**というものです。それが **brainbase** です。

今日は、「AIに働かされる」を実際に実装してみた立場から、柳川さんの議論に対するアンサーを書いてみます。

---

## 柳川さんの「ロングコンテキスト問題」に全面同意する

柳川さんの記事で最も重要な指摘は、**AIの致命的弱点は「ロングコンテキスト（長期間の文脈保持）」**だという点です。

> AIは「瞬間風速」は出せますが、「文脈（コンテキスト）をつなぎ続けること」ができません。過去の経緯、複雑な利害関係、未来へのプライオリティ……これらを保持し続け、整合性を取る能力が圧倒的に欠けているのです。

これ、激しく同意します。僕も最初、Claude Codeや各種AIエージェントを使い始めたとき、「爆速で出力してくれるけど、文脈忘れすぎ問題」にずっと悩まされました。

プロジェクトAの話をしてたのに、気づいたらプロジェクトBの前提で回答してる。昨日決めたことを今日は忘れてる。関係者の名前と役割がごちゃ混ぜになる。

これでは「人月AI」として機能しません。

---

## 僕のアプローチ：コンテキストを外部化する

柳川さんは「人間がコンテキストの保持者として君臨する」と書いています。

僕はこれに一つ付け加えたい。

**「コンテキストを人間の脳の外に出して、AIが参照できる形で構造化しておく」**

これが brainbase の設計思想です。

### なぜ外部化するのか？

理由は単純で、**僕一人が4事業のコンテキストを脳内で保持するのは無理**だからです。

- SalesTailor（AIパーソナライズセールスレター）
- Zeims（AI税務リサーチ）
- BAAO（生成AI導入コンサル）
- Tech Knight（ホテル業界DX）

これらの事業ごとに、戦略、KPI、進行中のタスク、決定事項、関係者、契約条件、ブランドガイドライン……全部覚えておくのは人間業じゃない。

かといって、AIに丸投げすると文脈がリセットされる。

だから、**「人間が保持すべきコンテキスト」を構造化してAIに渡せる形にしておく**ことにしました。

---

## brainbaseの3層アーキテクチャ

brainbaseは以下の3層で構成されています。

```
┌─────────────────────────────────────────────┐
│            L3: CEO Agent                    │
│  全プロジェクトのKPIダッシュボード          │
│  AI PMへの指示・優先度調整                  │
│  ユーザー（創業者）との対話インターフェース │
└─────────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
┌─────────────┐┌─────────────┐┌─────────────┐
│   AI PM     ││   AI PM     ││   AI PM     │
│ SalesTailor ││   Zeims     ││ TechKnight  │
└─────────────┘└─────────────┘└─────────────┘
                    │
┌─────────────────────────────────────────────┐
│        L1: mana (Slack Gateway)             │
│  Slackイベント受信・ルーティング           │
└─────────────────────────────────────────────┘
                    │
┌─────────────────────────────────────────────┐
│           brainbase 運用OS                  │
│  _codex/ (ナレッジ・判断基準・RACI)         │
│  Airtable (タスク・マイルストーン)          │
│  S3 (Slack履歴) / グラフiti (知識グラフ)     │
└─────────────────────────────────────────────┘
```

柳川さんの言う「人月AI管理者」の役割を、**AIエージェント自身が担う構造**です。

ただし、ここが重要なのですが、**AIエージェントがロングコンテキストを自力で保持しているわけではありません**。

brainbaseという「外部記憶」にアクセスすることで、擬似的にロングコンテキストを再現しているのです。

## 「意志」の外部化が核心

柳川さんは「意思決定ではなく意志が大事」と書いています。

これも完全に同意します。そして僕は、**意志すらも外部化できる**と考えています。

brainbaseには `_codex` というディレクトリがあります。ここに格納されているのは、

- **MVV（ミッション・ビジョン・バリュー）**
- **判断基準**
- **運用ルール**
- **RACI（責任分担）**
- **ブランドガイドライン**

つまり、「佐藤圭吾がどういう基準で判断するか」「何を大事にしているか」「どういう行動を良しとするか」が全て言語化されています。

AIエージェントは、単発の指示に従うのではなく、**この「意志の正本」を参照しながら動く**。

だから、僕が寝ている間でも、休日でも、AIエージェントは「佐藤ならこう判断するだろう」という行動を取れる。

これが「AIに働かされる」の本質だと思っています。

## 実際にやってみてわかったこと

### 1. 仕組み化が先、AI活用が後

柳川さんも「変革期に表面的なHowだけを追って思考停止で生きていたら詰む」と書いています。

brainbaseを構築してわかったのは、**AIを活用する前に、自分の仕事を徹底的に構造化する作業が必要**ということです。

僕が作った「90日仕組み化チェックリスト」は以下の通り：

- 01_strategy.md（戦略骨子）
- 02_offer/（価格・オファー設計）
- 03_sales_ops/（営業オペレーション）
- 04_delivery/（デリバリー手順）
- 05_kpi/（KPI定義）
- RACI定義
- ブランドガイド

これらが揃って初めて、AIエージェントに「この事業を回して」と言える。

逆に言えば、**この構造化をサボった状態でAIに丸投げしても、ハルシネーションの嵐になるだけ**です。

### 2. 人間の仕事は「週次30分レビュー」に収束する

brainbaseのゴール状態は、**創業者が週次30分レビューだけで事業を回せる状態**です。

これは柳川さんの言う「全体設計者」に近いですが、もう一段踏み込んでいます。

全体設計者の仕事の大半すらもAIに移譲し、人間は「異常値のアラートを受けて判断する」「大きな方針転換を決める」「人間同士の関係性を調整する」ことに特化する。

実際、僕は今、各事業のAI PMから上がってくるレポートをチェックして、「これはOK」「これは要確認」「これは人間が介入」を判断するだけの生活になりつつあります。

### 3. 「コンテキスト注入」がボトルネックになる

柳川さんは「AIのアウトプット単位が大きくなっても、人間がボトルネックになる」と指摘しています。

僕の経験では、このボトルネックは**「コンテキスト注入のタイミング」**に集中します。

新しいプロジェクトを立ち上げるとき、新しいメンバーが入ったとき、大きな方針転換があったとき。

これらのタイミングで、人間が brainbase に「意志」を注入する必要がある。

逆に言えば、**一度コンテキストを注入してしまえば、日常運用はほぼAIで回る**。

## 柳川さんの組織図に対するアンサー

柳川さんは「全体設計者」「深掘り職人」「人月AI管理者」「PMO/監査役」という役割分類を提示しました。

僕はこれを見て思ったのは、**これらの役割の大半はAIエージェント化できる**ということです。

| 柳川さんの分類 | brainbaseでの実装 |
|---------------|-------------------|
| 全体設計者 | CEO Agent + 人間（週次30分） |
| 深掘り職人 | Research Sub-agent |
| 人月AI管理者 | AI PM（プロジェクト単位） |
| PMO/監査役 | 異常検知アラート + 人間（必要時のみ） |

つまり、**人間が担うべき領域は「意志の注入」「異常時の判断」「人間関係の調整」の3つに収束する**と考えています。

これは「人間が不要になる」という話ではありません。

むしろ、**人間にしかできないことに人間が集中できる**という話です。

## 「AIに働かされる」の真の意味

柳川さんは「AIに働かされる」ことに抵抗がある人もいるだろうと書いています。

僕の解釈は少し違います。

**「AIに働かされる」とは、「AIが最大効率で動けるように人間が環境を整える」こと**です。

これは「下請け」ではありません。

むしろ、**監督・演出家に近い**。

映画監督は自分で演技しません。でも、役者が最高の演技をできるように、脚本を渡し、演出をつけ、環境を整える。

brainbaseは、AIエージェントという「役者」に渡す「脚本」であり「演出ノート」です。

---

## 結論：ロングコンテキスト問題の解は「外部化」

柳川さんの記事で最も重要な問いは、「ロングコンテキスト問題をどう解決するか」でした。

僕の答えは明確です。

**「人間の脳に保持するのではなく、外部化して構造化し、AIが参照できるようにする」**

これを実現する基盤が brainbase です。

もちろん、brainbase が唯一の解ではありません。各社・各人が自分なりの「外部脳」を構築すればいい。

ただ、**「AI活用の前に構造化」**という順序は間違いなく正しいと確信しています。
