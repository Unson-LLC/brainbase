# 100日チャレンジ - 毎日の進捗報告ワークフロー

**期間**: 2026-01-01 〜 2026-04-10（100日間）
**投稿頻度**: 毎日1回
**タグ**: `#100日チャレンジ`、`#AI事業自動化`

---

## 基本フロー

### Step 1: 今日自動化した業務を選択

**参照ファイル**: `_codex/sns/100day_challenge_tasks.md`

1. リストから1つ選択（未完了の業務）
2. または、リストにない新しい業務を追加

**確認事項**:
- [ ] その業務は本当に自動化されたか？
- [ ] Before/Afterの時間削減が明確か？
- [ ] 再現性のある仕組みか？（1回きりのスクリプトではない）

---

### Step 2: ドラフト作成

**推奨構文**: **進捗報告型**（ストーリー不要）

**テンプレート**:
```
【100日チャレンジ Day X】

今日自動化した業務:
「○○○」

Before: ××分かかっていた
After: ボタン1つで完了

仕組み:
- [具体的な自動化方法を1-2行で]

累計: X個/100個完了
残り: XX日

次は「△△△」に挑戦します。
```

**ポイント**:
- シンプル・具体的・再現性
- Before/Afterの時間削減を明示
- 「仕組み」は具体的に（ツール名・スクリプト名等）
- 次の予告で継続性を作る

---

**画像の役割**: 進捗の可視化

### 100日チャレンジ専用フロー

ユーザーが「100日チャレンジ」と明示した場合:

1. `100day_challenge_tasks.md`から未完了業務を表示
2. AskUserQuestionで選択を求める
3. 進捗報告型のドラフトを自動生成（Phase 2をスキップ）
4. 画像生成（progressテンプレート）
5. 投稿実行
6. タスクリスト自動更新

## 注意事項

### 投稿タイミング
- **毎日同じ時間帯**に投稿（例: 朝8時、夜9時）
- 投稿後3分はアクティブを保つ（リプ返、いいね等）

### ネタ被り防止
- 同じ業務を2回投稿しない
- `post_log.md`に自動記録される
- `100day_challenge_tasks.md`の進捗表で管理

### 失敗・挫折・遅延の扱い（重要）

**失敗・遅延は投稿対象であり、むしろプロセスエコノミーの核心**

✅ **投稿すべきケース**:
1. 「今日は自動化できなかった」
   - 例: 「Day X: タスク優先度判定の自動化に挑戦したけど、ロジック化できず断念」
2. 「実装したけど使い物にならなかった」
   - 例: 「Day X: 議事録自動生成を実装したけど、精度30%で使えない。明日リトライ」
3. 「前提条件が未整備で実装できない」
   - 例: 「Day X: FAQ自動応答を作りたいけど、FAQベースがまだない。先にFAQ整備が必要」
4. 「CEO Agent待ちで実装遅延」
   - 例: 「Day X: 戦略立案の自動化はCEO Agent（2月末実装予定）が前提。今は無理」

✅ **失敗投稿のテンプレート**:
```
【100日チャレンジ Day X】

今日挑戦した業務:
「○○○の自動化」

結果: 失敗 / 遅延 / 保留

理由:
- [具体的な原因を1-2行で]

学び:
- [この失敗から得た気づき]

累計: X個/100個完了（変わらず）
残り: XX日

次のアプローチ:
「△△△を試してみます」
```

✅ **失敗・遅延が多い場合の対処**:
- 「100日で100個」→「100日で挑戦100回」に目標をシフト
- 完了数ではなく、「挑戦回数」「学び」を重視
- SNS投稿では「累計: X個完了、Y個挑戦中」と表記

✅ **プロセスエコノミー的な価値**:
- 「完璧な自動化」より「試行錯誤の過程」の方が共感を生む
- 「CEO Agent待ち」という技術的制約を見せることで、brainbaseの設計思想が伝わる
- 「失敗→学び→再挑戦」のサイクルが、視聴者の記憶に残る

### 継続性の担保
- 1日でも抜けると「100日チャレンジ」が崩れる
- あらかじめ3-5日分のストックを作る
- または、「2日分まとめて報告」も許容範囲
