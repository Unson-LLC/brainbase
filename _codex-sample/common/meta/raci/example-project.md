# Example Project - RACI マトリックス

プロジェクトにおける責任分担を定義します。

## 📖 RACI定義

- **R (Responsible)**: 実行責任者 - 実際に作業を行う
- **A (Accountable)**: 説明責任者 - 最終的な責任を持つ（1名のみ）
- **C (Consulted)**: 相談先 - 意見を求められる
- **I (Informed)**: 報告先 - 結果を知らされる

## 📊 意思決定フロー

### 戦略的意思決定（A: Alice）

- プロダクトの方向性
- 新機能の追加・削除
- ビジネスモデル変更

**プロセス**:
1. Alice が Bob に相談 (C)
2. Alice が最終決定 (A)
3. Bob に結果を通知 (I)

### 技術的意思決定（A: Bob）

- 技術スタック選定
- アーキテクチャ設計
- パフォーマンス最適化

**プロセス**:
1. Bob が Alice に相談 (C)
2. Bob が最終決定 (A)
3. Alice に結果を通知 (I)

## 🔄 エスカレーション

### レベル1: 日常業務
- 各担当者が独自に判断
- 必要に応じて相談 (C)

### レベル2: プロジェクト影響
- 対応する A (Accountable) が判断
- チーム内で議論

### レベル3: 組織影響
- Alice（PM）が CEO に相談
- 経営陣で最終決定

## 📝 補足

### チーム拡大時

プロジェクトが成長し、メンバーが増えた場合:

1. デザイナー追加 → UI/UX設計の R
2. QAエンジニア追加 → テストの R
3. マーケター追加 → 顧客対応の R

各役割に応じて RACI を更新します。
